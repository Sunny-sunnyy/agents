Section 4

Core Theory - Lý thuyết cốt lõi

Docx LLM

Key concepts - Khái niệm chính

  • State - trạng thái: Snapshot ảnh chụp toàn bộ dữ liệu của hệ thống tại một thời điểm thực thi.
  • immutable - bất biến: Tính chất bảo đảm dữ liệu của State sau khi tạo ra không thể bị sửa đổi tùy tiện.
  • Annotated - kiểu chú thích mở rộng: Cú pháp chuẩn của Python dùng để đính kèm thêm metadata (như hàm reducer) vào gợi ý kiểu dữ liệu (type hint).
  • reducer - hàm hợp nhất cập nhật trạng thái: Hàm chịu trách nhiệm chỉ định cách thức LangGraph gộp các trường dữ liệu cập nhật mới từ Node vào State chung.
  • add_messages - reducer mặc định cho tin nhắn: Hàm reducer tích lũy có sẵn giúp tự động nối các đối tượng tin nhắn mới (hoặc dict tin nhắn) vào lịch sử tin nhắn cũ.
  • Pydantic BaseModel: Lớp cơ sở dùng để định nghĩa State class có cấu trúc, hỗ trợ kiểm tra kiểu dữ liệu tự động.

Concept relationships - Mối quan hệ giữa các khái niệm

Trong LangGraph, mối quan hệ điều phối dữ liệu diễn ra như sau:

Node là một hàm xử lý (transformer). Nó nhận một bản sao của State hiện tại. Sau khi chạy logic, Node trả về một tập hợp các thay đổi (dưới dạng State mới hoặc dictionary).

LangGraph đọc thông tin metadata khai báo tại Annotated của State class để tìm hàm Reducer tương ứng của trường đó, thực hiện state merging (hợp nhất trạng thái) một cách an toàn. Điều này cực kỳ quan trọng khi có parallel nodes (các nút chạy song song) cùng trả về kết quả cập nhật mà không ghi đè mất dữ liệu của nhau.

Deep explanation - Giải thích sâu về cơ chế hoạt động của add_messages

Hàm add_messages không chỉ là một phép cộng danh sách (list concatenation) đơn thuần. Bên dưới lớp vỏ, nó thực hiện các xử lý thông minh sau:

  • Tự động ép kiểu (Type coercion): Nếu Node của bạn trả về tin nhắn dưới dạng dict thuần, ví dụ: {"role": "user", "content": "hello"}, add_messages sẽ tự động chuyển nó thành đối tượng tin nhắn chuẩn của LangChain (như HumanMessage).
  • Xử lý cập nhật tin nhắn theo ID: Nếu tin nhắn mới trả về có trùng id với một tin nhắn đã tồn tại trong State, add_messages sẽ tiến hành ghi đè/cập nhật nội dung của tin nhắn cũ bằng nội dung mới thay vì nối thêm. Đây là cơ sở để cập nhật tin nhắn trong giao diện thời gian thực hoặc sửa lỗi tin nhắn.
Section 5

Workflow / Pipeline - Quy trình xây dựng Graph

Docx LLM Code

High-level flow - Quy trình 5 bước thực tế

  1. Bước 1: Định nghĩa lớp State: Kế thừa từ BaseModel (hoặc TypedDict) và khai báo các trường dữ liệu kết hợp cú pháp Annotated để gắn hàm reducer (ví dụ: add_messages).
  2. Bước 2: Khởi tạo Graph Builder: Tạo đối tượng quản lý cấu trúc đồ thị bằng lệnh StateGraph(State).
  3. Bước 3: Tạo và đăng ký các Nodes: Viết các hàm logic (nhận state cũ, trả về state mới) và đăng ký qua graph_builder.add_node(name, func).
  4. Bước 4: Thiết lập các Edges: Nối các node bằng add_edge(), sử dụng hằng số START để định nghĩa điểm vào và END để định nghĩa điểm ra của luồng chạy.
  5. Bước 5: Biên dịch (Compile) đồ thị: Gọi graph_builder.compile() để khóa cấu trúc đồ thị lại và chuyển thành một thực thể thực thi được.

Data / control flow - Luồng chạy khi gọi invoke()

Khi chạy đồ thị bằng lệnh graph.invoke(initial_state):

1. Tham số đầu vào được đưa qua hằng số kết nối START.
2. Dữ liệu đi thẳng vào Node đầu tiên trong luồng điều khiển.
3. Node đọc dữ liệu từ State, thực thi xử lý (ghép chuỗi ngẫu nhiên hoặc gọi API LLM), tạo ra một đối tượng State mới chứa kết quả trả về.
4. LangGraph nhận kết quả, gọi hàm reducer (ví dụ: add_messages) để tích lũy câu trả lời mới vào State chính.
5. Luồng chạy đi qua kết nối tiếp theo dẫn tới hằng số kết thúc END và trả về kết quả State cuối cùng cho chương trình gọi.

Section 6

Techniques - Kỹ thuật sử dụng

Immutable update pattern
(Mẫu cập nhật bất biến)

Purpose: Đảm bảo tính nhất quán của trạng thái, giữ nguyên lịch sử snapshot để phục vụ cho debug hoặc quay lại các bước trước (time-travel).

When to use: Bắt buộc áp dụng trong tất cả các Node functions. Tránh hành vi mutate trực tiếp thuộc tính của state đầu vào.

Trade-off: Tốn công sức khởi tạo đối tượng State mới (hoặc trả về dict thay đổi) thay vì gán giá trị trực tiếp như lập trình OOP thông thường.

Annotated metadata
(Khai báo metadata qua Annotated)

Purpose: Đính kèm hướng dẫn hợp nhất dữ liệu (reducer) trực tiếp tại cấu trúc khai báo dữ liệu của State class thay vì nhúng rải rác trong các Node.

When to use: Sử dụng khi khai báo các trường danh sách tin nhắn hoặc các trường cần tích lũy dữ liệu theo thời gian.

Trade-off: Đòi hỏi lập trình viên phải làm quen và học cú pháp Annotated của Python type system.

Gradio interface wrapper
(Lớp bọc giao diện Gradio)

Purpose: Tạo giao diện chat trực quan nhanh chóng để tương tác thử nghiệm với đồ thị trong file Jupyter Notebook.

When to use: Sử dụng ở pha dựng prototype nhanh để kiểm tra hành vi phản hồi của Agent.

Trade-off: Nếu không thiết kế cẩn thận, dễ làm mờ ranh giới giữa logic giao diện (UI) và logic điều phối của đồ thị (Graph).

Phần 1: Đồ thị tối giản không dùng LLM (Silly Node Graph)

Đoạn mã này minh họa cách xây dựng đồ thị cơ bản 5 bước hoàn chỉnh nhưng node xử lý chỉ ghép chữ ngẫu nhiên từ danh sách danh từ và tính từ, chứng minh LangGraph hoạt động độc lập và không phụ thuộc vào bản chất của mô hình ngôn ngữ lớn.

Jupyter Cell 1-10 Imports & Setup Random string generator 
from typing import Annotated
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
from pydantic import BaseModel
import random

# Định nghĩa tập dữ liệu mẫu dùng cho silly node
nouns = ["Cabbages", "Unicorns", "Toasters", "Penguins", "Bananas", "Zombies", "Rainbows", "Eels", "Pickles", "Muffins"]
adjectives = ["outrageous", "smelly", "pedantic", "existential", "moody", "sparkly", "untrustworthy", "sarcastic", "squishy", "haunted"]

# Bước 1: Định nghĩa lớp State sử dụng Pydantic BaseModel
# Sử dụng Annotated để gắn reducer add_messages vào trường messages
class State(BaseModel):
    messages: Annotated[list, add_messages]

# Bước 2: Khởi tạo Graph Builder với State class vừa định nghĩa
graph_builder = StateGraph(State)

# Bước 3: Tạo Node xử lý (là một hàm Python nhận old_state và trả về new_state)
def our_first_node(old_state: State) -> State:
    # Logic ghép danh từ và tính từ ngẫu nhiên
    reply = f"{random.choice(nouns)} are {random.choice(adjectives)}"
    messages = [{"role": "assistant", "content": reply}]
    
    # Tạo đối tượng State mới chứa câu trả lời và trả về (Immutable update)
    new_state = State(messages=messages)
    return new_state

# Đăng ký node vào builder với tên định danh "first_node"
graph_builder.add_node("first_node", our_first_node)

# Bước 4: Tạo Edges nối từ điểm START vào first_node, và từ first_node ra điểm END
graph_builder.add_edge(START, "first_node")
graph_builder.add_edge("first_node", END)

# Bước 5: Biên dịch đồ thị thành đối tượng có thể chạy được
graph = graph_builder.compile()

# Hàm chat bọc ngoài để chạy đồ thị với giao diện Gradio
def chat(user_input: str, history):
    # Mỗi lần chat được kích hoạt, tạo một state mới chỉ chứa tin nhắn hiện tại của người dùng
    message = {"role": "user", "content": user_input}
    messages = [message]
    state = State(messages=messages)
    
    # Gọi thực thi đồ thị
    result = graph.invoke(state)
    print("Kết quả State trả về:", result)
    
    # Trả về nội dung tin nhắn cuối cùng trong danh sách để hiển thị lên chat giao diện
    return result["messages"][-1].content

Phần 2: Chatbot thực tế tích hợp mô hình OpenAI và Gradio

Mã nguồn dưới đây thay thế node ngẫu nhiên bằng một node gọi mô hình gpt-4o-mini thông qua LangChain OpenAI wrapper. Luồng chạy vẫn giữ nguyên cấu trúc đồ thị tuyến tính.

Jupyter Cell 11-13 ChatOpenAI Integration Gradio Interface 
from langchain_openai import ChatOpenAI
import gradio as gr

# Bước 1: Định nghĩa lại State Class
class State(BaseModel):
    messages: Annotated[list, add_messages]

# Bước 2: Khởi tạo lại Graph Builder
graph_builder = StateGraph(State)

# Khởi tạo mô hình ngôn ngữ lớn (LLM) gpt-4o-mini
# API key được tự động lấy từ môi trường sau khi gọi load_dotenv()
llm = ChatOpenAI(model="gpt-4o-mini")

# Bước 3: Tạo Node gọi LLM thật sự
def chatbot_node(old_state: State) -> State:
    # Gọi mô hình LLM với toàn bộ lịch sử tin nhắn hiện có trong old_state
    response = llm.invoke(old_state.messages)
    
    # Trả về đối tượng State mới chứa tin nhắn phản hồi của LLM
    new_state = State(messages=[response])
    return new_state

# Đăng ký chatbot node vào builder
graph_builder.add_node("chatbot", chatbot_node)

# Bước 4: Tạo Edges liên kết
graph_builder.add_edge(START, "chatbot")
graph_builder.add_edge("chatbot", END)

# Bước 5: Biên dịch đồ thị
graph = graph_builder.compile()

# Hàm chat điều phối chạy đồ thị kết nối giao diện Gradio
def chat(user_input: str, history):
    # CHÚ Ý BẪY: Mỗi lần người dùng gửi tin nhắn, một State mới được tạo lập từ đầu
    # Chỉ chứa tin nhắn mới nhất, do đó đồ thị hiện tại không có bộ nhớ (Stateless)
    initial_state = State(messages=[{"role": "user", "content": user_input}])
    
    # Invoke đồ thị để LLM trả lời
    result = graph.invoke(initial_state)
    print("Trạng thái sau invoke:", result)
    
    # Trả về nội dung phản hồi cuối cùng của bot
    return result['messages'][-1].content

# Khởi chạy giao diện chat trực quan trong notebook
gr.ChatInterface(chat, type="messages").launch()

Options / Trade-offs - Bản đồ lựa chọn

Lựa chọn - Option Phù hợp nhất - Best for Ưu điểm - Pros Nhược điểm & Đánh đổi - Cons & Trade-off
Pydantic BaseModel làm State Khi làm việc trong môi trường Python lớn hoặc Jupyter Notebook cần kiểu dữ liệu cấu trúc chặt chẽ. Khai báo rõ ràng, hỗ trợ tự động validation dữ liệu đầu vào và đầu ra của từng trường. Nặng hơn một chút so với dict thuần, cú pháp khởi tạo cần import thư viện Pydantic.
TypedDict làm State Khi xây dựng các ứng dụng nhẹ, ưu tiên tốc độ xử lý hoặc giao tiếp nhanh với các API dạng JSON. Gần gũi với kiểu dictionary mặc định của Python, cú pháp khởi tạo cực kỳ đơn giản và nhanh gọn. Không hỗ trợ validation dữ liệu tự động tại runtime, khó quản lý các trường phức tạp.
Stateless per-invoke chatbot (Chatbot không lưu trạng thái) Làm các ứng dụng demo nhanh, kiểm tra kết nối API LLM và kiểm thử hoạt động của các nút logic đơn lẻ. Cấu trúc cực kỳ đơn giản, không cần cấu hình bộ lưu trữ cơ sở dữ liệu hoặc checkpointing. Chatbot hoàn toàn bị "mất trí nhớ" qua từng lượt hội thoại. Mỗi câu chat là một phiên độc lập.
Stateful memory chatbot (Chatbot lưu trạng thái) Các ứng dụng chatbot thực tế, trợ lý ảo cần ghi nhớ ngữ cảnh cuộc trò chuyện của người dùng. Tự động tích lũy lịch sử trò chuyện, duy trì ngữ cảnh dài hạn giúp trả lời chính xác và thông minh. Đòi hỏi cấu hình thêm cơ chế lưu trữ bền vững (checkpointing/persistence) và quản lý thread ID.

Pitfalls - Lỗi / bẫy thường gặp

Failure modes - Trường hợp dễ fail

  • Sửa đổi (Mutate) state cũ tại chỗ: Viết mã sửa trực tiếp list của old_state.messages bên trong hàm Node. Lỗi này làm mất đi tính snapshot của StateGraph, phá hỏng khả năng debug lịch sử đồ thị.
  • Hiểu lầm add_messages tự động duy trì bộ nhớ dài hạn của app: Nghĩ rằng vì State có reducer add_messages nên chatbot sẽ tự động nhớ tên người dùng ở lượt chat sau. Thực chất, mỗi lần Gradio gọi hàm chat, một đối tượng State mới tinh được tạo lập nên lịch sử cũ bị xóa sạch.
  • Khai báo gợi ý kiểu dữ liệu (type hint) thiếu Annotated: Chỉ khai báo messages: list trong State. Khi Node trả về tin nhắn mới, LangGraph sẽ ghi đè hoàn toàn danh sách cũ thay vì cộng dồn (append) do thiếu chỉ định hàm reducer.

Fix / prevention - Cách tránh

  • Viết code an toàn (Immutable): Luôn tạo một đối tượng State mới bằng cách trả về dictionary chứa thay đổi, ví dụ: return {"messages": [new_msg]} để framework tự gộp thay vì thay đổi đối tượng cũ.
  • Tách biệt Session và Invoke: Hiểu rằng reducer chỉ giúp hợp nhất tin nhắn trong phạm vi một lần gọi invoke() đồ thị. Để chatbot nhớ tin nhắn dài hạn, cần cấu hình thêm bộ lưu trữ trạng thái (memory persistence) bằng checkpointer ở các bài học sau.
  • Kiểm tra cú pháp Annotated kĩ lưỡng: Đảm bảo luôn viết messages: Annotated[list, add_messages] để kích hoạt cơ chế tích lũy tin nhắn tự động của LangGraph.
Section 10

Flashcards / Q&A - Thẻ ôn tập

Q: Tại sao State trong LangGraph lại được xem là bất biến (immutable)?

A: Vì State đại diện cho ảnh chụp hệ thống (snapshot) tại một thời điểm cụ thể. Tính bất biến giúp ngăn chặn các tác dụng phụ ngầm (side effects) khi các node chạy song song hoặc ghi đè dữ liệu lên nhau, đồng thời phục vụ cho việc quay lại các bước trước (time-travel debug).

Q: annotated đóng vai trò gì trong định nghĩa State của LangGraph?

A: Cú pháp Annotated cho phép đính kèm hàm reducer (metadata) trực tiếp vào gợi ý kiểu dữ liệu (type hint) của từng trường trong State class. LangGraph đọc metadata này để biết cách hợp nhất (merge) dữ liệu mới trả về từ Node vào State chung.

Q: Tại sao chatbot trong Lab 1 bị "mất trí nhớ" qua các lượt trò chuyện mặc dù đã dùng add_messages?

A: Vì mỗi lượt nhập tin nhắn mới trên Gradio, hàm chat bọc ngoài lại khởi tạo lại từ đầu đối tượng State bằng lệnh: initial_state = State(messages=[...]) chỉ chứa tin nhắn hiện tại. Lịch sử của lượt invoke() trước đó không được lưu trữ hoặc truyền lại vào lần gọi mới.

Q: Sự khác nhau cơ bản giữa compile() và invoke() trong LangGraph là gì?

A:
  • compile(): Là bước kết thúc pha định nghĩa đồ thị (Build phase), biên dịch cấu trúc đã khai báo thành một đối tượng thực thi.
  • invoke(): Là pha chạy thực tế (Execution phase), nhận vào trạng thái khởi đầu cụ thể và kích hoạt các Node chạy để trả về State kết quả.

Quick Memory - Ôn nhanh cuối bài

Must remember - Phải nhớ

  1. State = Snapshot bất biến. Tránh tuyệt đối việc gán/mutate state cũ trực tiếp bên trong node.
  2. Cú pháp Annotated: messages: Annotated[list, add_messages] là xương sống của các chatbot graph.
  3. Hàm Reducer: Giải quyết bài toán state merging an toàn, đặc biệt là khi có các node chạy song song (parallelism).
  4. Bot mất trí nhớ: Chatbot trong Lab 1 hiện tại là stateless giữa các lượt chat do khởi tạo State mới mỗi lần Gradio gọi hàm chat.
