Section 2

Cheat Sheet - Tóm tắt một trang

Core idea - Ý chính

LangGraph là một framework đồ thị chuyên dụng trong hệ sinh thái LangChain, thiết kế để xây dựng các agentic workflows - luồng tác vụ tác tử phức tạp có vòng lặp (loops). Nó mang lại tính ổn định, chịu lỗi và khả năng tái lập cho hệ thống agent ở cấp độ vận hành thực tế (production).

Must know - Cần nhớ

  • Tránh nhầm lẫn: Platform chỉ là hạ tầng deploy trả phí; việc lập trình thực tế được thực hiện hoàn toàn trên thư viện mã nguồn mở Framework.
  • Khái niệm cốt lõi: Graph = State (bộ nhớ chung) + Nodes (hàm thực thi logic) + Edges (đường nối điều phối luồng).
  • Hoài nghi lành mạnh: Abstraction frameworks giúp tăng tốc phát triển ban đầu nhưng có thể che giấu logic sâu bên dưới, làm tăng debugging cost (chi phí debug).

Use in practice - Dùng khi nào

Nên chọn LangGraph khi hệ thống cần: vòng lặp phản hồi (feedback loops), khả năng lưu trạng thái để khôi phục khi lỗi (checkpointing/persistence), hoặc có bước phê duyệt từ con người (human-in-the-loop). Đối với luồng tuần tự đơn giản, gọi API trực tiếp là baseline tốt nhất.

Previous Context - Liên hệ với bài trước

Nối tiếp các tuần học trước về OpenAI Agents SDK, CrewAI và các design patterns - mẫu thiết kế cho AI Agents. Các framework trước tập trung vào các chuỗi bước khá tuyến tính (chains) hoặc để các agent tự do hội thoại mà không có cấu trúc kiểm soát chặt chẽ.

LangGraph yêu cầu một bước chuyển đổi tư duy quan trọng: chuyển từ tư duy tuyến tính sang kiến trúc dựa trên đồ thị (graph-based architecture). Việc định nghĩa rõ ràng cấu trúc luồng chạy và quản lý trạng thái tập trung giúp đưa kỷ luật kỹ thuật (engineering discipline) vào phát triển Agent.
Section 4

Core Theory - Lý thuyết cốt lõi

Docx LLM

Key concepts - Khái niệm chính

  • LangChain - framework trừu tượng hóa cho ứng dụng LLM: Hệ sinh thái cung cấp các abstractions cho model calls, prompt templates, memory và RAG.
  • LangGraph - framework đồ thị cho agent workflows: Thư viện mã nguồn mở giúp tổ chức workflow agent dưới dạng đồ thị có trạng thái.
  • LangSmith - công cụ quan sát và debug: Sản phẩm độc lập để theo dõi reasoning traces, logs gọi model và debug lỗi hệ thống agent.
  • State - trạng thái: Snapshot dữ liệu sống dùng chung chảy qua toàn bộ graph, là chất mang dữ liệu trung tâm.
  • Node (nút): Python function nhận trạng thái hiện tại, thực thi logic (gọi LLM, chạy tool), và trả về một phần trạng thái được cập nhật.
  • Edge (cạnh): Định nghĩa đường nối giữa các Node để điều phối luồng chạy (gồm cạnh thường và cạnh có điều kiện).
  • Reducer - bộ hợp nhất cập nhật trạng thái: Cơ chế định nghĩa cách gộp các bản cập nhật mới từ Node vào State chung.

Concept relationships - Mối quan hệ giữa các khái niệm

State (Trạng thái) là trái tim của Graph. Khi một Node (Nút) được kích hoạt, nó nhận current state (trạng thái hiện tại) làm tham số đầu vào. Sau khi thực hiện logic, Node trả về các thông tin mới cần cập nhật.

Thông tin này đi qua Reducer (Bộ hợp nhất) để cập nhật một cách an toàn vào State chung. Edges (Cạnh) sẽ đọc dữ liệu từ State sau khi cập nhật để đưa ra quyết định kích hoạt Node tiếp theo (điều hướng).

Deep explanation - Giải thích sâu về tư duy State Immutable

LangGraph nhấn mạnh việc thiết kế và cập nhật trạng thái theo kiểu không thay đổi trực tiếp (immutable-style updates). Thay vì viết mã để sửa đổi trực tiếp các thuộc tính của đối tượng State bên trong node, các Node luôn nhận một snapshot của state cũ và trả về một dictionary mô tả những phần cần thay đổi.

Framework sẽ tự chịu trách nhiệm áp dụng các thay đổi này thông qua Reducer. Cách tiếp cận này giúp:

  • Ngăn chặn triệt để lỗi side effects (tác dụng phụ) do các node vô tình ghi đè dữ liệu của nhau.
  • Hỗ trợ tính năng time-travel debug (quay ngược thời gian debug) - cho phép xem lại chính xác snapshot của State tại từng bước chạy trong lịch sử đồ thị.
  • Tạo cơ sở cho cơ chế khôi phục lỗi (checkpointing) khi có sự cố mạng hoặc lỗi API trong quá trình gọi LLM.
Section 5

Workflow / Pipeline - Quy trình xây dựng Graph

Docx LLM

High-level flow - Luồng 5 bước xây dựng

  1. Định nghĩa State Class: Xác định schema dữ liệu dùng chung (ví dụ: dùng TypedDict chứa danh sách messages).
  2. Khởi tạo Graph Builder: Tạo đối tượng builder (ví dụ: StateGraph(StateClass)) để bắt đầu vẽ đồ thị.
  3. Tạo các Nodes: Viết các hàm Python logic (nhận state, trả về cập nhật) rồi đăng ký vào builder qua add_node().
  4. Tạo các Edges: Nối các nút lại bằng add_edge() cho các luồng cố định hoặc add_conditional_edges() cho rẽ nhánh điều kiện.
  5. Biên dịch (Compile) Graph: Gọi builder.compile() để chuyển khai báo đồ thị thành một executable (thực thể có thể chạy được).

Data / control flow - Luồng dữ liệu và điều khiển

Khi bắt đầu chạy ứng dụng graph bằng lệnh graph.invoke(initial_state):

1. Dữ liệu ban đầu (initial state) đi vào điểm khởi đầu START.
2. Framework chuyển tiếp dữ liệu đến Node đầu tiên theo cấu hình của Edge kết nối với START.
3. Node chạy logic, trả về dữ liệu cập nhật, State chung được cập nhật thông qua Reducer.
4. Edge tiếp theo được đánh giá. Nếu là conditional edge, một hàm router sẽ được gọi để phân tích State hiện tại và chọn tên Node đích tiếp theo.
5. Quy trình lặp lại cho đến khi luồng điều khiển đi vào điểm kết thúc END.

Section 6

Techniques - Kỹ thuật sử dụng

State-first design
(Thiết kế ưu tiên State)

Purpose: Xác định trước cấu trúc dữ liệu chung mà toàn bộ hệ thống Agent cần sử dụng trước khi viết bất kỳ code xử lý logic nào.

When to use: Bắt buộc áp dụng ngay từ pha thiết kế ban đầu của mọi dự án LangGraph để tránh phá vỡ kiến trúc dữ liệu về sau.

Trade-off: Đòi hỏi thời gian phân tích và lập kế hoạch kỹ lưỡng trước khi bắt tay vào gõ code logic node.

Functional nodes
(Xây dựng nút dạng hàm)

Purpose: Giữ cho mã nguồn của từng bước xử lý độc lập, dễ viết unit test và không gây ra tác dụng phụ ngầm lên biến toàn cục.

When to use: Áp dụng cho mọi hàm đăng ký làm Node. Hàm chỉ nên tương tác với thế giới bên ngoài thông qua State đầu vào và trả về.

Trade-off: Yêu cầu kỷ luật lập trình cao, tránh thói quen thay đổi trực tiếp (mutate) các biến bên ngoài hoặc đối tượng truyền vào.

Conditional routing
(Định tuyến có điều kiện)

Purpose: Cho phép hệ thống Agent tự đưa ra quyết định rẽ nhánh luồng chạy linh hoạt dựa trên phân tích ngữ cảnh của LLM.

When to use: Khi quy trình làm việc có các điểm quyết định (ví dụ: LLM phân tích xem code có lỗi không, nếu có thì rẽ sang node sửa code, nếu không thì trả kết quả).

Trade-off: Làm tăng độ phức tạp của đồ thị, đòi hỏi viết thêm các hàm router phụ trợ để phân tích State.

Tạo Đồ Thị Có Vòng Lặp Cơ Bản (Minimal Stateful Graph)

Mã nguồn dưới đây xây dựng một đồ thị LangGraph đơn giản gồm: một State chứa tin nhắn, một Node chatbot phản hồi, một Node đánh giá kết quả, và một Edge có điều kiện để mô phỏng vòng lặp sửa lỗi.

Python 3.10+ Library: langgraph >= 0.1.0 Key concepts: StateGraph, Reducer, Router 
from typing import Annotated, TypedDict
from langgraph.graph import StateGraph, START, END

# Định nghĩa hàm reducer để chỉ rõ cách cập nhật danh sách tin nhắn (nối thêm vào danh sách cũ)
def append_messages(left: list[str], right: list[str]) -> list[str]:
    return left + right

# Bước 1: Define state class (Định nghĩa cấu trúc trạng thái chung)
class AgentState(TypedDict):
    # Sử dụng Annotated để khai báo reducer append_messages cho trường messages
    messages: Annotated[list[str], append_messages]
    needs_review: bool

# Bước 2: Start graph builder (Khởi tạo bộ dựng đồ thị với State class)
builder = StateGraph(AgentState)

# Bước 3: Create nodes (Viết các Python function đóng vai trò là các nút xử lý)
def chatbot_node(state: AgentState):
    print("--- CHẠY NODE: Chatbot xử lý câu hỏi ---")
    # Lấy tin nhắn cuối cùng để xử lý (giả lập kết quả gọi LLM)
    user_msg = state["messages"][-1]
    bot_response = f"Bot: Đã nhận phản hồi cho câu hỏi '{user_msg}'"
    
    # Trả về dictionary chứa các trường muốn cập nhật vào State chung
    # LangGraph sẽ tự dùng reducer để nối bot_response vào messages
    return {
        "messages": [bot_response],
        "needs_review": True # Kích hoạt cờ kiểm tra đánh giá ở bước sau
    }

def review_node(state: AgentState):
    print("--- CHẠY NODE: Đánh giá chất lượng câu trả lời ---")
    # Giả lập logic kiểm tra: Nếu đây là phản hồi thứ hai của bot, duyệt thông qua. 
    # Nếu là phản hồi đầu tiên, yêu cầu bot tạo lại (giả lập feedback loop)
    bot_responses = [m for m in state["messages"] if m.startswith("Bot:")]
    
    if len(bot_responses) < 2:
        print("   -> Đánh giá: Câu trả lời chưa đủ sâu. Yêu cầu làm lại.")
        return {"needs_review": True, "messages": ["System: Hãy cải thiện câu trả lời"]}
    else:
        print("   -> Đánh giá: Đạt chất lượng. Kết thúc.")
        return {"needs_review": False}

# Đăng ký các Node vào builder
builder.add_node("chatbot", chatbot_node)
builder.add_node("reviewer", review_node)

# Bước 4: Create edges (Thiết lập các kết nối và điều phối luồng)
# START kết nối trực tiếp vào chatbot
builder.add_edge(START, "chatbot")

# Từ chatbot đi thẳng tới reviewer (unconditional edge)
builder.add_edge("chatbot", "reviewer")

# Viết hàm Router để quyết định rẽ nhánh dựa trên trạng thái (State)
def route_after_review(state: AgentState):
    # Nếu cờ review vẫn bằng True, quay lại chatbot để sửa (tạo vòng lặp)
    # Nếu cờ review bằng False, kết thúc đồ thị (đi tới END)
    if state["needs_review"]:
        return "chatbot"
    return END

# Khai báo conditional edge từ node reviewer sang các node đích tiếp theo dựa trên router
builder.add_conditional_edges(
    "reviewer",
    route_after_review,
    {
        "chatbot": "chatbot",
        END: END
    }
)

# Bước 5: Compile graph (Biên dịch đồ thị thành thực thể thực thi)
graph = builder.compile()

# Chạy thử nghiệm Graph với trạng thái khởi tạo
if __name__ == "__main__":
    initial_input = {
        "messages": ["User: Giải thích về LangGraph"],
        "needs_review": False
    }
    
    print("=== BẮT ĐẦU CHẠY ĐỒ THỊ ===")
    final_output = graph.invoke(initial_input)
    print("\n=== KẾT QUẢ CUỐI CÙNG TRONG STATE ===")
    for idx, msg in enumerate(final_output["messages"]):
        print(f" Bước {idx + 1}: {msg}")

Section 9

Pitfalls - Lỗi / bẫy thường gặp

Failure modes - Trường hợp dễ fail

  • Nhầm lẫn thương mại: Cho rằng phải trả phí cho LangGraph Platform thì mới sử dụng được LangGraph.
  • Over-abstraction (Trừu tượng hóa quá mức): Ép buộc sử dụng LangGraph cho mọi tác vụ gọi LLM, kể cả những luồng tuyến tính cực kỳ đơn giản chỉ có 1-2 bước.
  • Mutate state trực tiếp trong Node: Thay đổi trực tiếp các giá trị trong đối tượng State nhận vào thay vì trả về dictionary thay đổi.
  • Tư duy tuyến tính ẩn: Khai báo cấu trúc graph đồ thị nhưng thực chất chỉ nối các node chạy một lèo từ đầu đến cuối mà không có rẽ nhánh hay lặp lại.

Fix / prevention - Cách tránh

  • Hiểu rõ bối cảnh: Khóa học và việc lập trình thực tế chạy trên thư viện mã nguồn mở LangGraph Framework hoàn toàn miễn phí.
  • Giữ sự tối giản: Bắt đầu dự án bằng cách gọi trực tiếp API (Direct API usage) để kiểm soát prompt. Chỉ chuyển dịch sang LangGraph khi độ phức tạp của luồng (vòng lặp, human approval) đòi hỏi.
  • Tuân thủ tính Immutable: Luôn viết node nhận state cũ, thực hiện tính toán và trả về dict chứa giá trị thay đổi. Để framework tự gộp qua Reducer.
  • Vẽ sơ đồ luồng trước: Vẽ nháp sơ đồ hoạt động của agent ra giấy để xác định rõ các điều kiện rẽ nhánh (conditional edges) và điểm lặp (feedback loops) trước khi code.
Section 10

Flashcards / Q&A - Thẻ ôn tập

Q: LangGraph giải quyết vấn đề cốt lõi nào của LangChain?

A: LangChain được thiết kế cho các chuỗi bước tuyến tính (chains), rất khó tổ chức vòng lặp (loops) một cách ổn định. LangGraph giải quyết vấn đề này bằng cách đưa ra graph-based architecture - kiến trúc dựa trên đồ thị có trạng thái, giúp quản lý tốt các vòng lặp phản hồi (feedback loops) và lưu trạng thái để khôi phục khi lỗi (checkpointing).

Q: Sự khác nhau cơ bản giữa Node và Edge trong LangGraph là gì?

A:
  • Node (Nút): Là Python function chứa logic xử lý thực tế (ví dụ: gọi LLM, lưu file). Nhận state và trả về bản cập nhật.
  • Edge (Cạnh): Là kết nối điều hướng, quyết định luồng chạy sẽ đi từ Node nào sang Node nào tiếp theo dựa trên dữ liệu trong State.

Q: Tại sao LangGraph yêu cầu cập nhật State theo kiểu không thay đổi trực tiếp (immutable-style updates)?

A: Để tránh các tác dụng phụ ngầm (side effects) khi các node ghi đè dữ liệu lên nhau, đồng thời lưu trữ lịch sử các trạng thái phục vụ cho việc quay lại các bước trước (time-travel debug) và khôi phục lỗi (resiliency) qua checkpointing.

Q: 5 bước cơ bản để định nghĩa và thực thi một Graph trong LangGraph là gì?

A:
  1. Định nghĩa State class (schema dữ liệu chung).
  2. Khởi tạo Graph Builder (đối tượng StateGraph).
  3. Tạo các Nodes (hàm logic) và đăng ký vào builder.
  4. Tạo các Edges (kết nối giữa các node).
  5. Biên dịch (Compile) đồ thị để sẵn sàng thực thi qua phương thức invoke().

Quick Memory - Ôn nhanh cuối bài

Must remember - Phải nhớ

  1. LangGraph giúp đảm bảo ba tiêu chuẩn: stability - ổn định, resiliency - khả năng chịu lỗi, và repeatability - khả năng lặp lại nhất quán.
  2. LangGraph Framework là mã nguồn mở, miễn phí chạy local; Studio là công cụ trực quan; Platform là dịch vụ chạy cloud có phí.
  3. Mô hình chạy gồm hai pha riêng biệt: Khai báo cấu trúc (Build) và Biên dịch chạy thực tế (Compile & Invoke).
  4. Tránh thay đổi trực tiếp (mutate) state trong node. Hãy để framework gộp thông qua Reducer.
