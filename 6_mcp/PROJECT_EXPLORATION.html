Từ các thực hành khóa học MCP đến một autonomous trading floor - sàn giao dịch tự động trực tiếp.

Đợt khám phá này tái cấu trúc 6_mcp như một systems atlas - bản đồ hệ thống cho việc gia nhập: dự án là gì, các module (mô-đun) liên kết với nhau như thế nào, state (trạng thái) thời gian chạy luân chuyển ở đâu, và cách dự án capstone project - dự án cuối khóa chuyển đổi lý thuyết MCP thành một runnable multi-agent product demo - bản mô tả sản phẩm đa tác tử có thể chạy được.

Được căn cứ trực tiếp trên 5 files (tập tin) HTML study guide - Hướng dẫn học tập, 15 modules (mô-đun) Python gốc và 5 notebooks (sổ tay lập trình) gốc. Setup notes remain inference-based - Các ghi chú thiết lập vẫn dựa trên suy luận vì đợt xem xét này tập trung vào việc đọc và thiết kế repo (kho chứa), chứ không phải thực thi thời gian chạy.

Lộ trình khóa học

Ngày 1 tiêu thụ MCP. Ngày 2 hiển thị logic nghiệp vụ. Ngày 3 mở rộng các bề mặt tính năng. Ngày 4-5 hợp nhất mọi thứ thành một hệ thống giao dịch có thể quan sát được.

Chế độ vận hành

Các stdio MCP servers - máy chủ MCP stdio local-first - ưu tiên cục bộ, SQLite-backed state - trạng thái hỗ trợ bởi SQLite, các external APIs - giao diện lập trình ứng dụng bên ngoài tùy chọn, các quy trình UI (giao diện người dùng) và trading-loop processes - quy trình vòng lặp giao dịch riêng biệt.

Tại sao nó quan trọng

Bài học thực sự không phải là tài chính. Đó là cấu trúc tác tử mô-đun: resources (tài nguyên), prompts (lời nhắc), orchestration - sự điều phối, xem xét trace (dấu vết), và sản phẩm hóa.

Mẫu thiết kế đặc trưng

Researcher-as-tool - nhà nghiên cứu như một công cụ giúp một nhà giao dịch duy trì quyền kiểm soát trong khi vẫn cho phép cắm tính năng nghiên cứu chuyên biệt vào một cách sạch sẽ.

Tổng quan dự án

Một repo (kho chứa), hai nhiệm vụ: dạy về MCP và xây dựng một capstone project - dự án cuối khóa hệ thống đáng tin cậy.

Dự án được hiểu tốt nhất như một phương tiện giảng dạy tiến dần từ các bản demo MCP riêng lẻ đến một hệ thống dùng chung duy nhất: bốn trader agents - tác tử giao dịch, một researcher capability - khả năng nghiên cứu, shared account state - trạng thái tài khoản dùng chung, và một observability layer - lớp quan sát được mà bạn thực sự có thể kiểm tra.

Những gì dự án này đang giải quyết

6_mcp giải quyết câu hỏi "làm thế nào để các phần này trở thành một hệ thống thống nhất?" thường vẫn còn mơ hồ sau các bài hướng dẫn về MCP. Thay vì chỉ dừng lại ở các bản demo server (máy chủ), nó chỉ ra cách tools (công cụ), resources (tài nguyên), biểu mẫu prompt (lời nhắc), account state - trạng thái tài khoản, tracing - ghi dấu vết, và UI (giao diện người dùng) có thể được kết hợp thành một runtime loop - vòng lặp thời gian chạy cụ thể.

Sản phẩm cuối cùng là một sàn giao dịch mô phỏng, nhưng giá trị lâu dài là khả năng hiểu biết về kiến trúc: hiểu được ranh giới giữa domain state - trạng thái miền, MCP surface - bề mặt MCP, orchestration - sự điều phối, và khả năng quan sát.

Mục tiêu: component (thành phần) đa tác tử có thể tái sử dụng Chế độ: sự kết hợp MCP local-first - ưu tiên cục bộ Đầu ra: dashboard (bảng điều khiển) + loop (vòng lặp) + traces (dấu vết)

Module weave - Sự đan xen các khối

Xét về mặt phụ thuộc, dự án không phải là một đống kịch bản phẳng. Nó có một trục chính mạnh mẽ: từ bộ lập lịch đến orchestration - sự điều phối nhà giao dịch, từ nhà giao dịch đến các bản khai báo và prompts (lời nhắc) MCP, sau đó là dịch vụ tài khoản và thị trường bên dưới. UI (giao diện người dùng) nằm riêng một góc và đọc cùng một trạng thái được lưu trữ thay vì triển khai lại logic thời gian chạy.

Scheduler entry - Đầu vào bộ lập lịch

trading_floor.py tạo ra 4 đối tượng nhà giao dịch, đính kèm LogTracer - bộ theo dõi nhật ký, kiểm soát hành vi mở cửa thị trường và chạy vòng lặp vô tận.

Trader orchestration - Điều phối nhà giao dịch

traders.py xây dựng tác tử, đóng gói nhà nghiên cứu dưới dạng một công cụ, tìm nạp các resource (tài nguyên) tài khoản và quản lý vòng đời của server (máy chủ) MCP.

Prompt composition - Cấu trúc prompt (lời nhắc)

templates.py và mcp_params.py giữ cho văn bản prompts (lời nhắc) và các khai báo máy chủ nằm ngoài logic điều phối.

Service bridge - Cầu nối dịch vụ

accounts_client.py làm cầu nối giữa các resource (tài nguyên) và tools (công cụ) MCP stdio thành các hàm có thể gọi và các định nghĩa công cụ thân thiện với SDK.

Domain + persistence - Miền + Bền vững

accounts.py sở hữu số dư, danh mục nắm giữ, giao dịch và logic PnL; database.py lưu trữ bền vững tài khoản, logs (nhật ký) và cache (bộ nhớ đệm) thị trường.

UI observer - Người quan sát UI (giao diện người dùng)

app.py sử dụng cùng dữ liệu tài khoản và log (nhật ký) để kết xuất một dashboard (bảng điều khiển), nhờ đó màn hình hiển thị luôn dựa trên trạng thái thời gian chạy thực tế.

Runtime loop - Vòng lặp chạy thực tế

Vòng lặp này là trái tim hành vi của dự án. Mỗi lượt đi qua sẽ luân phiên giao dịch và tái cân bằng, nhưng mẫu sâu hơn là: làm đầy resource (tài nguyên) trước, sử dụng tool (công cụ) thứ hai, lưu trữ trạng thái bền vững thứ ba, và khả năng quan sát xuyên suốt.

1

Khởi tạo môi trường chạy

Bộ lập lịch xây dựng các nhà giao dịch và mở các context (ngữ cảnh) MCP stdio bằng AsyncExitStack.

2

Làm đầy trạng thái

Đọc báo cáo tài khoản và resource (tài nguyên) chiến lược trước khi mô hình bắt đầu đưa ra quyết định.

3

Cấu trúc prompts (lời nhắc)

Chèn trạng thái chạy thực tế vào các hướng dẫn giao dịch hoặc tái cân bằng từ templates.py.

4

Nghiên cứu

Nhà giao dịch gọi researcher tool - công cụ nghiên cứu, công cụ này có thể sử dụng fetch, Brave, memory (bộ nhớ) và context (ngữ cảnh) thị trường.

5

Thực thi

Các công cụ mua, bán, thay đổi chiến lược và thông báo đẩy đều khả dụng thông qua MCP.

6

Lưu trữ bền vững

Cập nhật tài khoản, logs (nhật ký) và chuỗi thời gian danh mục đầu tư được lưu trữ trong trạng thái hỗ trợ bởi SQLite.

7

Quan sát

Các đoạn trace (dấu vết) được ghi nhật ký, UI (giao diện người dùng) làm mới và chu kỳ tiếp theo bắt đầu sau khoảng thời gian chờ được cấu hình.

Setup path - Đường dẫn thiết lập

Mô hình thiết lập là local-first - ưu tiên cục bộ và định hướng giảng dạy. Bạn cấu hình các nhà cung cấp tính năng, tùy chọn đặt lại thông tin nhân vật, sau đó chạy UI (giao diện người dùng) và engine (công cụ thực thi) dưới dạng các quy trình riêng biệt.

1. Chuẩn bị môi trường

Cài đặt Python bằng uv. Cài đặt Node và npx nếu sử dụng các server (máy chủ) MCP dựa trên JavaScript như Brave Search hoặc Playwright.

uv --version npx --version

2. Cấu hình nhà cung cấp

Thiết lập các API keys - khóa API và các cờ như BRAVE_API_KEY, POLYGON_API_KEY, POLYGON_PLAN, và các khóa đa mô hình tùy chọn.

BRAVE_API_KEY=... POLYGON_API_KEY=... POLYGON_PLAN=paid|realtime

3. Chạy sàn giao dịch

Tùy chọn đặt lại các nhà giao dịch, sau đó khởi chạy dashboard (bảng điều khiển) và background loop - vòng lặp nền trong các cửa sổ dòng lệnh riêng biệt.

uv run reset.py uv run app.py uv run trading_floor.py

Vận hành

Cách chạy, các thành phần phụ thuộc và cách xác thực.

Không có bản khai báo phụ thuộc chính thức nào trong phạm vi được khám phá, vì vậy bức tranh vận hành đến từ các import (nhập khẩu) và các lệnh trong notebook (sổ tay lập trình) thay vì các tệp khóa phiên bản (lockfiles).

Hướng dẫn thiết lập và chạy

  • Cài đặt Python bằng uv.
  • Cài đặt Node và npx cho các server (máy chủ) MCP dựa trên JavaScript khi cần thiết.
  • Cấu hình các khóa như BRAVE_API_KEY, POLYGON_API_KEY, PUSHOVER_*, và các khóa nhà cung cấp mô hình tùy chọn.
  • Tùy chọn chạy uv run reset.py.
  • Khởi chạy uv run app.py và uv run trading_floor.py trong các cửa sổ dòng lệnh riêng biệt.

Ngăn xếp công nghệ và các thành phần phụ thuộc

  • Python + uv / uvx
  • OpenAI Agents SDK - Bộ công cụ phát triển phần mềm tác tử OpenAI
  • MCP Python SDK - Bộ công cụ phát triển phần mềm Python MCP / FastMCP
  • sqlite3, pydantic, python-dotenv, requests
  • polygon, gradio, pandas, plotly
  • Các server (máy chủ) MCP dựa trên Node cho Brave Search, memory (bộ nhớ), fetch và Playwright tùy chọn

Chiến lược thử nghiệm

  • Không tìm thấy bộ thử nghiệm chuyên dụng nào trong phạm vi được khám phá.
  • Phong cách xác thực hiện tại dựa trên notebook (sổ tay lập trình) và tập trung nhiều vào trace (dấu vết)/dashboard (bảng điều khiển).
  • Các thử nghiệm tiếp theo tốt nhất: unit tests - thử nghiệm đơn vị cho domain/state (miền/trạng thái), thử nghiệm hợp đồng MCP, và integration tests - thử nghiệm tích hợp cho Trader.run().
