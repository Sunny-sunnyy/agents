Section 2

Cheat Sheet / Executive Summary

Core idea - Ý chính

MCP - giao thức ngữ cảnh mô hình là chuẩn giúp agent hoặc host app kết nối với tool/resource/prompt do bên khác cung cấp. Giá trị lớn nhất không nằm ở “kỹ thuật mới” mà nằm ở việc tiêu chuẩn hóa cách chia sẻ tool để tạo ecosystem.

Must know - Cần nhớ

  • Protocol vs Framework: MCP không thay OpenAI Agents SDK, CrewAI hay LangGraph.
  • Local first: đa số MCP server chạy local trên máy bạn, thường qua Stdio.
  • SDK practice: MCPServerStdio + list_tools() + Agent(..., mcp_servers=[...]).
  • Security: dùng MCP server gần với chạy code bên thứ ba hơn là bật một “tính năng vô hại”.

Use in practice - Dùng khi nào

Khi agent của bạn cần năng lực phổ biến như browse web, đọc/ghi file, search, database access, hoặc các integration do cộng đồng đã xây sẵn, MCP giúp bạn tái sử dụng thay vì tự viết toàn bộ tool.

Knowledge extension - kiến thức mở rộng: nguồn tổng hợp nhắc rằng MCP còn có Tools, Resources, Prompts; trong ngày 1, trọng tâm gần như nằm hẳn ở Tools.

Previous Context - Liên hệ với bài trước

Week 6 quay lại OpenAI Agents SDK sau khi khóa học đã đi qua OpenAI Agents SDK, CrewAI, LangGraph và Autogen. Vì vậy Day 1 của MCP không bắt đầu từ con số 0: nó giả định bạn đã biết agent, tool, orchestration và khái niệm @function_tool. Điểm mới ở đây là không phải học cách “tạo tool cho riêng mình”, mà học cách dùng tool ecosystem qua một chuẩn giao tiếp chung.
Vì sao mạch này hợp lý?

Nếu chưa học agent framework trước đó, bạn sẽ rất dễ hiểu sai MCP là “framework mới”. Nhưng sau khi đã thấy framework lo phần orchestration và tools nội bộ, bạn mới phân biệt được rõ vai trò của MCP: nó là lớp kết nối bên ngoài, không phải lõi dựng agent.

Section 4

Core Theory - Lý thuyết cốt lõi

day1_summary.md lesson transcripts

Khái niệm nền

  • Model Context Protocol - giao thức ngữ cảnh mô hình: chuẩn hóa cách host hoặc agent dùng tool/resource/prompt từ bên ngoài.
  • Protocol vs Framework - giao thức so với framework: protocol định nghĩa cách nói chuyện; framework cung cấp bộ công cụ để xây hệ thống.
  • USB-C of Agentic AI - cổng kết nối chuẩn của AI agent: nhấn vào tính kết nối và tương thích, không nhấn vào việc “làm agent thông minh hơn”.
  • Tool ecosystem - hệ sinh thái công cụ: lợi ích lớn nhất của MCP là tái sử dụng hàng nghìn server/tool có sẵn.

Kiến trúc vận hành

  • MCP Host - ứng dụng chủ: Claude Desktop hoặc app agent do bạn viết bằng SDK.
  • MCP Client - thành phần kết nối: chạy trong host và nối 1-1 với từng MCP server.
  • MCP Server - máy chủ công cụ: chứa logic thật của tool, resource hoặc prompt.
  • Stdio - giao tiếp qua standard input/output: cách phổ biến nhất cho local server.
  • SSE - Server-Sent Events: phù hợp hơn với remote/hosted server.

Misconception dễ gặp

Từ “server” khiến nhiều người tưởng MCP server luôn chạy remote. Nguồn bài học nhấn mạnh điều ngược lại: đa số MCP server trong trải nghiệm developer hiện tại chạy local trên máy bạn, rồi từ đó có thể gọi web hoặc API ra ngoài.

Quan hệ giữa các phần

Framework như OpenAI Agents SDK lo phần agent loop, model call, trace, và orchestration. MCP lo phần “cắm thêm năng lực”. Nói ngắn gọn: framework xây agent; MCP trang bị tool cho agent.

Deep explanation - điểm khó nhất cần hiểu thật chắc

“MCP makes tool integration frictionless” không có nghĩa là MCP làm mọi thứ đơn giản hơn trong mọi trường hợp. Nếu bạn chỉ cần một tool nội bộ rất nhỏ, @function_tool hoặc function call thuần thường còn nhẹ hơn. MCP bắt đầu có lợi rõ rệt khi giá trị nằm ở chia sẻ, tái sử dụng, và chuẩn hóa tool metadata: description, parameters, transport, lifecycle. Đây là lý do giảng viên nhấn rất mạnh vào adoption hơn là novelty.

Knowledge extension - phần ngoài nguồn cần tách riêng

Bản tổng hợp có nhắc rằng spec công khai của MCP phổ biến hơn hẳn ở phần Tools, còn Resources và Prompts chưa được nhắc kỹ trong lab ngày 1. Khi tự học tiếp, bạn nên hiểu Day 1 này đang xây nền cho cách nghĩ “tool interoperability” trước, chưa phải bức tranh full spec.

  4. 112: chuyển sang server Node-based như Playwright và file system, làm quen npx.
  5. 113: gắn nhiều MCP server vào một agent để làm task thật.
  6. 114: nhìn ra bức tranh ecosystem qua marketplace và security trade-off.

Luồng kỹ thuật khi agent dùng MCP

  1. Chuẩn bị params mô tả command + args để spawn MCP server.
  2. Dùng MCPServerStdio(params=...) để tạo client session với server local.
  3. SDK có thể gọi list_tools() để biết server cung cấp tool nào.
  4. Tạo Agent(..., mcp_servers=[...]) để agent nhìn thấy tập tool được gom từ các server.
  5. Runner.run() vận hành agent loop: model nghĩ, chọn tool, nhận kết quả, rồi quyết định bước kế tiếp.

Data / control flow - luồng dữ liệu / điều khiển

User request vào host app → host agent suy nghĩ → MCP client gọi đúng server → server thực thi tool thật → kết quả quay về agent → agent tổng hợp câu trả lời hoặc ghi file. Trong lab, Playwright lo lấy thông tin web; file system server lo ghi markdown xuống thư mục sandbox.

3 cấu hình nên phân biệt

  • Local server qua Stdio: phổ biến nhất trong dev workflow.
  • Remote server qua SSE: ít gặp hơn trong bài, nhưng quan trọng về mặt kiến trúc.
  • Local server nhưng gọi internet: dễ bị nhầm là “remote MCP server”, trong khi server vẫn chạy trên máy local.
Từ góc nhìn phỏng vấn: nên kể luồng này như thế nào?

Câu trả lời gọn nhất là: “Tôi dùng OpenAI Agents SDK làm host orchestration, dùng MCPServerStdio để spawn từng local MCP server, rồi truyền chúng vào agent qua mcp_servers. Model không cần biết tool đến từ server nào; SDK expose chúng như năng lực mà agent có thể dùng trong runtime.”

Section 6

Techniques - Kỹ thuật sử dụng

MCPServerStdio

Purpose: spawn local MCP server và mở client session qua stdin/stdout.

When to use: default choice cho local MCP workflows trong lab.

Trade-off: đơn giản và rõ vòng đời, nhưng không phải transport cho remote server.

list_tools() inspection

Purpose: xem server thực sự expose tool nào trước khi đưa vào agent.

When to use: lúc học server mới, debug integration, hoặc kiểm tra metadata tool.

Trade-off: thêm một bước kiểm tra, đổi lại giảm mơ hồ đáng kể.

Node server discovery

Purpose: dùng npx để chạy Playwright hoặc file system server từ ecosystem JavaScript.

When to use: khi server chuẩn cộng đồng nằm ở npm thay vì Python package.

Trade-off: cần môi trường Node/Playwright phù hợp, đổi lại tool ecosystem rất lớn.

Multi-server composition

Purpose: kết hợp nhiều năng lực, ví dụ browse web + write file trong cùng một task.

When to use: tác vụ end-to-end cần nhiều tool domains khác nhau.

Trade-off: tăng complexity của runtime và security surface.

Instruction shaping

Purpose: hướng agent xử lý cookie banner, retry, persistence khi automation web.

When to use: bất cứ khi nào tool có tính không ổn định như browser navigation.

Trade-off: prompt dài hơn, nhưng tỉ lệ hoàn thành task tốt hơn nhiều.

Security vetting

Purpose: xem publisher, GitHub, community, rating trước khi chạy server.

When to use: luôn luôn, nhất là khi tool đến từ bên thứ ba ít tên tuổi.

Trade-off: tốn thời gian ban đầu, giảm nguy cơ chạy code độc hại.

So với @function_tool, MCP khác ở “kỹ thuật sử dụng” nào?

@function_tool tối ưu cho tool bạn tự viết trong codebase của mình. MCP tối ưu cho tool portability - tính di động của công cụ và shared metadata - siêu dữ liệu dùng chung. Nếu bạn không cần share hay reuse tool ecosystem, MCP có thể là thêm overhead.

Code Walkthrough - Phân tích code

1_lab1.ipynb day1_summary.md
Buổi học có code được cung cấp, nên section này tập trung vào các block thật sự xuất hiện trong notebook: fetch server, Node-based servers, và multi-server agent.

Block 1 - Imports và primitive chính

Đây là khối xác định “mình sẽ dùng host framework nào và dùng MCP ở lớp nào”.

Purpose: khai báo Agent SDK + MCP bridge Key names: Agent, Runner, trace, MCPServerStdio 
from dotenv import load_dotenv
from agents import Agent, Runner, trace
from agents.mcp import MCPServerStdio
import os

load_dotenv(override=True)

# Ý chính:
# - Agent / Runner / trace đến từ OpenAI Agents SDK
# - MCPServerStdio là điểm nối sang MCP server local
# - load_dotenv() chỉ chuẩn bị môi trường, không phải logic MCP

Ghi nhớ rằng MCP ở đây không thay vai trò của SDK. SDK vẫn là nơi quản lý agent loop; MCPServerStdio chỉ mang tool capability vào loop đó.

Block 2 - Spawn fetch server và inspect tools

Lab không đẩy thẳng tool vào agent ngay, mà cho bạn nhìn tool inventory trước bằng list_tools().

Purpose: tạo local MCP client/session cho fetch server Key functions: MCPServerStdio(), list_tools() 
fetch_params = {"command": "uvx", "args": ["mcp-server-fetch"]}

async with MCPServerStdio(
    params=fetch_params,
    client_session_timeout_seconds=60
) as server:
    fetch_tools = await server.list_tools()

# Ý nghĩa:
# - params mô tả lệnh cần chạy để spawn server
# - dùng timeout 60s để tránh mặc định quá ngắn
# - list_tools() giúp nhìn rõ server cấp cho agent những gì

Điểm hay ở đây là metadata của tool đã được chuẩn bị sẵn bởi người viết server. Bạn không chỉ nhận “một hàm”, mà nhận cả mô tả giúp LLM biết khi nào nên dùng tool đó.

Block 3 - Node-based MCP servers: Playwright và file system

Đây là đoạn cho thấy ecosystem MCP không bị giới hạn trong Python.

Purpose: gọi server từ npm ecosystem Key details: npx, sandbox path, scoped file access 
playwright_params = {
    "command": "npx",
    "args": ["@playwright/mcp@latest"]
}

sandbox_path = os.path.abspath(os.path.join(os.getcwd(), "sandbox"))
files_params = {
    "command": "npx",
    "args": ["-y", "@modelcontextprotocol/server-filesystem", sandbox_path]
}

# Ý nghĩa:
# - Playwright server cho browser automation
# - file system server bị giới hạn vào sandbox_path
# - đây là ví dụ rõ của principle "least privilege - đặc quyền tối thiểu"

Phần file system rất đáng chú ý: thay vì cho agent đụng toàn bộ máy, notebook giới hạn nó vào thư mục sandbox. Đây là một security pattern nên giữ lại khi tự xây demo sau này.

Block 4 - Multi-server agent chạy task thật

Đây là “aha moment” của lab: agent dùng browser tools để tìm nội dung và file tools để ghi kết quả.

Purpose: compose nhiều MCP server trong một agent Key functions: Agent(), Runner.run(), trace() 
instructions = """
You browse the internet to accomplish your instructions.
You are highly capable at browsing the internet independently to accomplish your task,
including accepting all cookies and clicking 'not now' as appropriate.
If one website isn't fruitful, try another. Be persistent.
"""

async with MCPServerStdio(params=files_params, client_session_timeout_seconds=60) as mcp_server_files:
    async with MCPServerStdio(params=playwright_params, client_session_timeout_seconds=60) as mcp_server_browser:
        agent = Agent(
            name="investigator",
            instructions=instructions,
            model="gpt-4.1-mini",
            mcp_servers=[mcp_server_files, mcp_server_browser]
        )
        with trace("investigate"):
            result = await Runner.run(
                agent,
                "Find a great recipe for Banoffee Pie, then summarize it in markdown to banoffee.md"
            )

Hai ý phải nhớ: (1) agent và Runner.run() nằm bên trong context manager của server, nếu không server có thể đã đóng trước lúc tool được gọi; (2) instructions đóng vai trò cực lớn khi tool có tính thực nghiệm như browser automation.

Notebook này dạy ta điều gì ngoài cú pháp?

Quan trọng nhất là tư duy composition: bạn không “attach tool từng cái” như với ví dụ nhỏ của function calling, mà đang attach server capability bundles. Một server có thể mang nhiều tool; một agent có thể mang nhiều server; và SDK là lớp điều phối việc model chọn dùng tool nào vào lúc nào.

Section 9

Pitfalls - Lỗi / bẫy thường gặp

Failure modes - trường hợp dễ fail

  • Nhầm MCP là framework: dẫn đến kỳ vọng sai, nghĩ nó thay orchestration layer.
  • Hiểu “server” là remote mặc định: bỏ lỡ fact quan trọng rằng phần lớn lab chạy local.
  • Tạo agent ngoài context manager: server đóng sớm, tool call vỡ ở runtime.
  • Prompt quá sơ sài cho browser automation: agent bỏ cuộc khi gặp popup, cookie banner hoặc site đầu tiên không tốt.
  • Tin marketplace như app store an toàn mặc định: vô tình chạy code chưa vet kỹ.

Fix / prevention - cách tránh

  • Luôn diễn đạt MCP là protocol bổ sung cho framework.
  • Phân biệt rõ server chạy local với server gọi internet.
  • Đặt Agent() và Runner.run() bên trong các khối async with MCPServerStdio(...).
  • Viết instructions cụ thể về retry, cookies, persistence và behavior mong muốn.
  • Check publisher, repo, stars, community, security rating và phạm vi quyền truy cập trước khi chạy.
Section 10

Flashcards / Q&A - Thẻ ôn tập

Q: MCP là gì trong một câu?

A: MCP là Model Context Protocol - giao thức ngữ cảnh mô hình, một chuẩn kết nối để host hoặc agent dùng tool/resource/prompt bên ngoài theo cách nhất quán.

Q: Vì sao gọi là “USB-C of Agentic AI”?

A: Vì nó nhấn mạnh vai trò connectivity - khả năng kết nối giữa agent và tool, chứ không phải bản thân MCP tạo ra năng lực agent mới.

Q: Stdio thường dùng cho trường hợp nào?

A: Cho local MCP server, nơi host spawn process ngay trên máy local và nói chuyện qua standard input/output.

Q: Điểm học được lớn nhất từ block multi-server agent là gì?

A: Agent có thể dùng nhiều MCP server trong một nhiệm vụ thống nhất; SDK sẽ gom năng lực công cụ và để model quyết định lúc nào dùng cái nào.
