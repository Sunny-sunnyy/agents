Cheat Sheet / Executive Summary

Core idea - Ý chính

Day 3 dạy rằng MCP không chỉ là “một server”. Nó là một họ pattern tích hợp: local-only, local-call-cloud, và remote/provider-managed. Bài học vì thế chuyển trọng tâm từ “có tool nào?” sang “tool đó nằm ở đâu, ai vận hành, plan nào mở được, và agent có đủ khôn để gọi đúng không?”.

Must know - Cần nhớ

  • Memory trong ví dụ được triển khai như entity tools dùng cho persistent memory - bộ nhớ bền vững.
  • Brave Search MCP chứng minh server local vẫn có thể là gateway tới dữ liệu web mới.
  • Polygon free plan buộc wrapper phải tiết kiệm API calls bằng EOD cache - bộ nhớ đệm cuối ngày.
  • Polygon paid/realtime mở đường cho official MCP server nhưng làm tăng tool overload.

Use in practice - Dùng khi nào

Khi bạn thiết kế agent thật, Day 3 giúp bạn quyết định nhanh: memory nên là server local riêng theo user, web research nên qua provider MCP hay tool trực tiếp, và data provider nào nên dùng wrapper tối giản thay vì ném cả chục tool vào agent.

One-sentence summary - Tóm tắt một câu

MCP làm agent mạnh hơn nhờ capability reuse - tái sử dụng năng lực, nhưng Day 3 nhắc rất rõ rằng capability càng rộng thì yêu cầu về prompt, routing, feature flag và observability càng cao.

Interview angle - Góc phỏng vấn

Nếu được hỏi “khi nào nên build custom MCP, khi nào nên dùng official provider MCP?”, Day 3 cho bạn câu trả lời có chiều sâu: phụ thuộc vào plan, freshness của dữ liệu, số lượng tools, mức kiểm soát mong muốn, và cost (chi phí) vận hành.

Section 4

Core Theory - Lý thuyết cốt lõi

Core của Day 3 không nằm ở một API cụ thể, mà ở cách phân loại capability và ranh giới giữa local process, remote data source, và hosted provider surface. Đây là nền để bạn không dùng MCP theo kiểu “có gì gắn nấy”.

day3_summary.md 3_lab3.ipynb market.py

Three MCP deployment patterns - ba kiểu triển khai MCP

  • Local-only server - server local thuần: xử lý và lưu dữ liệu ngay trên máy người dùng, ví dụ memory server với file ./memory/ed.db.
  • Local server calling cloud API - server local gọi API đám mây: local process vẫn được spawn từ host, nhưng dữ liệu được kéo từ provider như Brave Search hoặc Polygon.
  • Provider-managed / remote MCP - MCP từ xa do provider vận hành: capability nằm ở phía nhà cung cấp, thường đi kèm auth và account constraints.

Memory as tools/context - memory như tools và context

Một ý rất đáng nhớ của Day 3 là memory không nên bị thần thánh hóa. Trong notebook, memory xuất hiện như một tập entity tools mà agent gọi khi muốn tạo entities, observations hoặc relationships. Nghĩa là memory là capability có cấu trúc, không phải “một hộp đen tự nhớ”.

Cách nhìn này khiến thiết kế agent modular - mô-đun hơn: muốn nhớ kiểu gì thì gắn tool tương ứng, muốn reset hay tách người dùng thì đổi file database tương ứng.

Knowledge graph memory - bộ nhớ dạng đồ thị tri thức

Knowledge graph memory lưu facts dưới dạng nodes và relationships thay vì chỉ append thêm raw text. Với dạng cấu trúc này, agent có thể truy xuất “Ed là ai”, “Ed dạy gì”, “MCP liên quan thế nào đến Ed” theo quan hệ, chứ không chỉ rà lại lịch sử chat dài.

Trong ví dụ Day 3, điều này được lộ ra rất rõ khi agent ở lượt sau có thể trả lời lại thông tin đã lưu nhờ tái dùng cùng memory store.

Plan-aware capability - capability phụ thuộc plan

Phần Polygon giới thiệu một nguyên tắc thiết kế thực tế: cùng là “market data”, nhưng tool surface thực sự dùng được phụ thuộc mạnh vào plan API. Free plan phải tránh rate limit bằng cách cache EOD prices; paid hoặc realtime plan mới đáng để mở official MCP server có nhiều tool hơn.

Vì vậy, “có official server” không tự động có nghĩa “nên dùng ngay”. Bạn vẫn phải hỏi: plan của mình có mở đủ endpoint chưa, agent có dễ lạc tool không, và cost (chi phí) có xứng đáng không?

Deep explanation - Vì sao Day 3 quan trọng hơn chuyện cài thêm tool?

Day 3 thực chất dạy bạn architecture literacy - năng lực đọc kiến trúc cho MCP. Cùng một yêu cầu “cho agent truy cập dữ liệu”, bạn có thể giải theo ít nhất ba cách khác nhau: lưu local, gọi cloud từ local wrapper, hoặc dùng provider MCP nguyên bản. Mỗi cách đổi boundary về bảo mật, độ mới của dữ liệu, độ khó debug, khả năng test, và hành vi routing của model. Đây là lý do Day 3 là cây cầu nối giữa việc học MCP như protocol và việc dùng MCP như hệ thống production thật.

Kiến thức mở rộng: trong production, cùng một business capability đôi khi được expose qua tiered tool surface - bề mặt công cụ theo tầng: agent rẻ dùng wrapper hẹp, agent đắt tiền hơn mới được dùng provider MCP rộng. Đây là cách kiểm soát cost (chi phí) và giảm sai tool.
Section 5

Workflow / Pipeline - Quy trình / luồng hoạt động

day3_summary.md 3_lab3.ipynb mcp_params.py

Flow 1 - Memory loop

  1. Notebook spawn mcp-memory-libsql qua MCPServerStdio với LIBSQL_URL=file:./memory/ed.db.
  2. Agent được nhắc rõ rằng entity tools là persistent memory - bộ nhớ bền vững của nó.
  3. User cung cấp facts về Ed, MCP, và khóa học.
  4. Agent ghi các facts đó vào graph thông qua memory tools.
  5. Ở lượt sau, cùng memory file được dùng lại và agent truy xuất thông tin đã lưu để trả lời “What do you know about me?”.

Flow 2 - Brave Search research loop

  1. Host spawn local Brave Search MCP server bằng npx -y @modelcontextprotocol/server-brave-search.
  2. Server local nhận BRAVE_API_KEY từ môi trường nhưng không lộ key cho guide.
  3. Agent nhận instructions để search web và tóm tắt ngắn.
  4. Request về tin Amazon stock được chuyển thành search actions tới Brave Search backend.
  5. Kết quả web được tổng hợp lại thành market outlook - góc nhìn thị trường cho câu hỏi người dùng.

Flow 3 - Free Polygon plan path

  1. Notebook kiểm tra POLYGON_API_KEY.
  2. market.py dùng Polygon API để lấy dữ liệu market.
  3. Vì free plan bị rate limit nặng, code không gọi snapshot liên tục mà kéo cả market EOD rồi cache vào database.
  4. market_server.py chỉ expose một tool hẹp: lookup_share_price(symbol).
  5. Agent vì thế có interface đơn giản, ít cơ hội gọi nhầm tool và ít phụ thuộc vào breadth của provider.

Flow 4 - Paid / realtime Polygon path

  1. Notebook cấu hình official MCP bằng uvx --from git+https://github.com/polygon-io/mcp_polygon@v0.1.0 mcp_polygon.
  2. Agent được cấp một tool surface rất rộng từ provider.
  3. Prompt phải ràng buộc rõ tool như get_snapshot_ticker vì plan $29 chưa mở hết mọi API.
  4. mcp_params.py dùng POLYGON_PLAN để chọn official server cho paid/realtime, hoặc local wrapper cho free.
  5. Capstone từ đây có thể nâng cấp market intelligence mà không phải viết lại toàn bộ kiến trúc agent.
Decision points - Điểm quyết định: Day 3 liên tục buộc bạn chọn giữa độ rộng capability và độ dễ kiểm soát. Memory local thì đơn giản và an toàn hơn; Brave Search thêm freshness nhưng cần key; official Polygon MCP mạnh hơn rất nhiều, nhưng đổi lại là plan constraints, tool overload, và due diligence.
Section 6

Techniques - Kỹ thuật sử dụng

Techniques của Day 3 phần lớn là kỹ thuật thiết kế hệ thống và điều khiển hành vi agent, không chỉ là mẹo code. Mỗi kỹ thuật dưới đây đều có dấu vết trực tiếp trong summary hoặc code.

Memory server per identity - tách memory theo định danh

Purpose: tránh trộn facts của nhiều người dùng hoặc nhiều agent vào cùng một graph.

When to use: khi app có nhiều persona, nhiều user, hoặc hội thoại dài qua nhiều phiên.

Trade-off: quản lý nhiều file/db hơn, nhưng đổi lại giảm cross-contamination - lẫn ngữ cảnh.

Instructional priming - định hướng dùng tool bằng instructions

Purpose: nói rõ cho model rằng entity tools là persistent memory, không để nó tự đoán vai trò của tool.

When to use: khi tool có semantics - ý nghĩa dùng cụ thể mà model dễ bỏ qua.

Trade-off: prompt dài hơn, nhưng routing ổn định hơn nhiều.

EOD cache strategy - chiến lược cache cuối ngày

Purpose: né rate limit của free Polygon plan bằng cách kéo grouped market data một lần rồi tái dùng.

When to use: khi dữ liệu không cần realtime và API quota hạn chế.

Trade-off: freshness giảm, nhưng performance và cost (chi phí) tốt hơn.

Thin wrapper server - server bọc mỏng

Purpose: giữ business logic ở market.py, còn MCP server chỉ làm lớp expose tool.

When to use: khi muốn wrapper dễ test, dễ đổi logic nền mà không động vào protocol nhiều.

Trade-off: ít flexibility hơn server “đa năng”, nhưng dễ reason hơn.

Prompt-constrained tool selection - ràng buộc chọn tool qua prompt

Purpose: ép model dùng đúng API theo plan, ví dụ get_snapshot_ticker.

When to use: khi provider server có quá nhiều tools hoặc plan không mở hết endpoint.

Trade-off: không bảo đảm 100%, nhưng giảm đáng kể khả năng lạc tool.

Plan flagging - gắn cờ theo plan

Purpose: cho phép cùng codebase chọn đúng market path bằng POLYGON_PLAN.

When to use: khi dự án phải chạy ở nhiều tier dữ liệu khác nhau.

Trade-off: thêm một lớp branching logic, nhưng upgrade path mượt hơn hẳn.

Technique insight - kỹ thuật nào là quan trọng nhất để mang sang project thật?

Nếu chỉ chọn một, hãy mang theo thin wrapper + plan flagging. Hai kỹ thuật này cho phép bạn bắt đầu với tool surface rất hẹp, kiểm soát hành vi agent tốt, rồi chỉ mở rộng khi plan hoặc nhu cầu thực sự tăng lên. Đây là cách tránh bị official integrations “nuốt chửng” quá sớm.

Block 1 - Memory server bootstrap trong 3_lab3.ipynb (cells 3-6)

Đây là chỗ Day 3 biến memory từ ý niệm thành local MCP server có persistence thật.

Purpose: tạo memory server, ghi facts, rồi chứng minh agent nhớ lại được Key names: MCPServerStdio, LIBSQL_URL, Runner.run() 
params = {
    "command": "npx",
    "args": ["-y", "mcp-memory-libsql"],
    "env": {"LIBSQL_URL": "file:./memory/ed.db"},
}

instructions = (
    "You use your entity tools as a persistent memory "
    "to store and recall information about your conversations."
)

async with MCPServerStdio(params=params, client_session_timeout_seconds=30) as mcp_server:
    agent = Agent(name="agent", instructions=instructions, model=model, mcp_servers=[mcp_server])
    result = await Runner.run(agent, "My name's Ed. What do you know about me?")

Điểm hay của block này là nó không “giả vờ thông minh”. Agent nhớ được vì có cùng ed.db được tái dùng, và vì instructions nói rõ entity tools đóng vai trò persistent memory. Nếu bỏ một trong hai yếu tố này, hành vi nhớ sẽ yếu đi đáng kể.

Block 3 - market.py: logic chọn nguồn giá và cache

File này là business logic thật cho market data free-plan path, và nó đáng học hơn wrapper server.

Purpose: chọn free/paid behavior và giảm số lần gọi API Key names: get_market_for_prior_date(), get_share_price_polygon_eod(), get_share_price_polygon_min() 
is_paid_polygon = polygon_plan == "paid"
is_realtime_polygon = polygon_plan == "realtime"

@lru_cache(maxsize=2)
def get_market_for_prior_date(today):
    market_data = read_market(today)
    if not market_data:
        market_data = get_all_share_prices_polygon_eod()
        write_market(today, market_data)
    return market_data

def get_share_price_polygon(symbol) -> float:
    if is_paid_polygon:
        return get_share_price_polygon_min(symbol)
    else:
        return get_share_price_polygon_eod(symbol)

Ý tưởng rất thực dụng: free plan không đủ “sang” để gọi dữ liệu tươi liên tục, nên module này đổi chiến lược sang lấy cả market EOD rồi cache. Chính quyết định này biến một API bị rate limit thành một capability còn dùng được cho agent.

Block 4 - market_server.py: lớp MCP mỏng

Wrapper này cực kỳ mỏng, và đó là chủ ý thiết kế đúng.

Purpose: expose một tool hẹp cho agent Key names: FastMCP, @mcp.tool(), lookup_share_price() 
from mcp.server.fastmcp import FastMCP
from market import get_share_price

mcp = FastMCP("market_server")

@mcp.tool()
async def lookup_share_price(symbol: str) -> float:
    return get_share_price(symbol)

if __name__ == "__main__":
    mcp.run(transport="stdio")

Toàn bộ business complexity nằm ở market.py; server chỉ là packaging layer - lớp đóng gói. Cách chia này làm Day 3 nhất quán với Day 2: wrapper mỏng, domain logic riêng, protocol không ôm việc thừa.

Block 5 - mcp_params.py: nối Day 3 vào capstone

Đây là block quan trọng để thấy Day 3 đã chảy vào multi-agent project path.

Block này cho thấy rõ “học xong là dùng ngay”: trader và researcher không nhận cùng một bộ tools. Researcher có fetch, Brave Search và memory; trader có accounts, push và market. Đây là tư duy capability partitioning - phân tách năng lực theo vai trò.

Section 9

Pitfalls - Lỗi / bẫy thường gặp

Failure modes - trường hợp dễ fail

  • Xem memory như phép màu: quên rằng agent chỉ nhớ tốt khi có đúng tools, đúng file store và đúng instructions.
  • Gom mọi user vào một memory DB: dẫn đến lẫn facts và sai ngữ cảnh qua phiên.
  • Dùng official MCP quá sớm: có nhiều tool nhưng model chưa đủ ổn để route đúng.
  • Bỏ qua rate limit free plan: gọi market data kiểu naive rồi làm hệ thống chậm hoặc fail.
  • Không gắn plan flag: code và tài khoản thực tế không khớp nhau, khiến agent dùng sai path.
  • Tin repo GitHub là auto-safe: chạy code third-party mà thiếu review nguồn.

Fix / prevention - cách tránh

  • Luôn ghi rõ semantics của tool trong instructions nếu tool đó đóng vai trò đặc biệt như memory.
  • Tách memory theo tên hoặc identity, ví dụ ./memory/{name}.db.
  • Bắt đầu với wrapper hẹp trước khi mở full provider tool surface.
  • Dùng cache, grouped fetch, hoặc batch strategy khi provider free tier bị quota thấp.
  • Biến plan thành feature flag rõ ràng như POLYGON_PLAN, không để logic ngầm.
  • Review repo, publisher, mức độ official, issues và release path trước khi chạy MCP từ GitHub.
