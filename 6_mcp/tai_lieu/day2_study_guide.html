Cheat Sheet / Executive Summary

Core idea - Ý chính

Day 2 dạy rằng MCP server không làm business logic của bạn tốt hơn; nó làm logic đó dễ công bố, dễ chia sẻ, và dễ dùng lại hơn. Vì vậy, quyết định viết MCP server trước hết là một quyết định về abstraction và distribution, không phải về “code cho xịn”.

Must know - Cần nhớ

  • 115: shareability là lý do số 1 để chọn MCP.
  • 116: accounts.py mới là lõi domain; server chỉ bọc nó ra ngoài.
  • 117: native MCP integration là đường hiện đại; handwritten client chủ yếu để học internals.
  • 118: sau khi hiểu plumbing, nên quay lại dùng abstraction và ecosystem cho đúng đích.

Use in practice - Dùng khi nào

Khi bạn có capability đủ đáng giá để nhiều host hoặc nhiều agent cùng cần, việc đóng gói bằng MCP giúp nó reusable hơn. Ngược lại, nếu tool chỉ sống trong đúng một app, @function_tool thường kinh tế hơn.

Knowledge extension - kiến thức mở rộng: summary Day 2 có nhắc hybrid strategy: giữ một số tool ở local function, còn capability nào muốn plug-and-play hoặc chia sẻ thì mới nâng lên MCP.

Section 4

Core Theory - Lý thuyết cốt lõi

day2_summary.md accounts*.py

Producer mindset - tư duy người tạo capability

  • Packaging layer - lớp đóng gói: server bọc business logic hiện có để công bố ra ngoài theo chuẩn MCP.
  • Shareability - khả năng chia sẻ: lý do mạnh nhất để chấp nhận thêm process, metadata, transport và lifecycle.
  • Internal tool vs shared tool - tool nội bộ vs tool chia sẻ: tiêu chí quyết định có nên dùng MCP hay không.
  • Extra plumbing - phần tích hợp bổ sung: process riêng, session/client setup, schema, transport, resources, spawning.

Server và client theo Day 2

  • FastMCP server: cách nhanh để expose tool/resource từ Python code.
  • Tool - công cụ: các hành động có side effect hoặc trả về dữ liệu, ví dụ buy_shares, get_balance.
  • Resource - tài nguyên: dữ liệu đọc theo URI, ví dụ accounts://accounts_server/{name}.
  • Handwritten client - client viết tay: giúp hiểu lifecycle như initialize(), list_tools(), call_tool(), read_resource().

Business logic mới là tài sản chính

Day 2 nhấn mạnh rằng phần khó và phần giá trị không nằm ở decorator @mcp.tool(). Giá trị thật nằm ở domain model và hành vi: balance, holdings, transaction, strategy, persistence, pricing. MCP chỉ biến phần đó thành capability có thể dùng bởi host khác.

Abstraction choice - chọn mức trừu tượng

Nếu mục tiêu là tốc độ cho một app đơn lẻ, native tool call thường hợp lý hơn. Nếu mục tiêu là consistency, portability hoặc multi-host reuse, MCP trở nên hấp dẫn. Đây là “build vs wrap” decision trọng tâm của Day 2.

Deep explanation - điều khó nhất của Day 2

Cái bẫy lớn nhất là tưởng “viết MCP server” đồng nghĩa với “đang nâng cấp tool của mình”. Thực ra bạn đang thay đổi cách công bố tool, không tự động nâng cấp chất lượng của bản thân tool đó. Nếu logic domain nghèo nàn, MCP chỉ giúp bạn chia sẻ một capability nghèo nàn ra ngoài theo chuẩn đẹp hơn. Vì vậy Day 2 cố tình tái dùng code tài khoản/chứng khoán đã có sẵn thay vì viết toy example từ đầu.

Knowledge extension - phần ngoài nguồn cần tách riêng

Summary gợi ý rằng nhiều team production sẽ không “all-in MCP” hay “all-in local function”. Họ dùng chiến lược lai: các tool rất nội bộ để local, còn capability cần tái sử dụng giữa IDE assistant, backend agent và desktop host thì mới bọc thành MCP server.

Section 6

Techniques - Kỹ thuật sử dụng

Problem framing before implementation

Purpose: quyết định xem tool này cần shareability hay chỉ cần local function call.

When to use: trước khi bọc bất kỳ business logic nào thành server.

Trade-off: tốn thêm bước suy nghĩ, đổi lại tránh overengineering.

Expose domain via FastMCP

Purpose: công bố các method/domain behaviors thành MCP tools và resources.

When to use: khi bạn đã có domain model đủ tốt và muốn bọc nó theo protocol.

Trade-off: API viết nhanh, nhưng vẫn cần business logic chất lượng phía sau.

URI-based resources

Purpose: phục vụ context đọc được qua địa chỉ chuẩn, ví dụ account report hoặc strategy.

When to use: khi dữ liệu phù hợp kiểu “readable context” hơn là action tool.

Trade-off: agent không phải lúc nào cũng tự tận dụng resource tốt như tool nếu client layer không hỗ trợ rõ.

Native SDK integration

Purpose: để OpenAI Agents SDK lo phần client/tool aggregation thay cho bạn.

When to use: mặc định trong đa số use case ngày nay.

Trade-off: nhanh và gọn, nhưng bạn thấy ít internals hơn.

Handwritten client

Purpose: học lifecycle, custom integration, resource-centric flows và schema bridge.

When to use: debug, học protocol, hoặc cần adapter layer đặc thù.

Trade-off: nhiều plumbing code, dễ lặp lại những gì SDK làm sẵn.

Schema bridge to FunctionTool

Purpose: chuyển MCP tool metadata sang interface tools mà OpenAI agent đã quen dùng.

When to use: khi bạn cần compatibility với tooling flow cũ hoặc custom wrappers.

Trade-off: phải xử lý schema shape và invocation mapping bằng tay.

Technique đáng nhớ nhất của Day 2 là gì?

Không phải một decorator cụ thể, mà là cách nghĩ: “MCP server as a packaging layer - MCP server như lớp đóng gói”. Khi tư duy này rõ, bạn sẽ biết lúc nào cần server, lúc nào chỉ cần local function.

Block 2 - accounts_server.py: bọc domain thành tools và resources

Đây là chỗ Day 2 biến domain logic thành capability chia sẻ được qua protocol.

Purpose: expose actions và context theo chuẩn MCP Key names: FastMCP, @mcp.tool(), @mcp.resource(), mcp.run() 
from mcp.server.fastmcp import FastMCP
from accounts import Account

mcp = FastMCP("accounts_server")

@mcp.tool()
async def get_balance(name: str) -> float:
    return Account.get(name).balance

@mcp.tool()
async def buy_shares(name: str, symbol: str, quantity: int, rationale: str) -> float:
    return Account.get(name).buy_shares(symbol, quantity, rationale)

@mcp.resource("accounts://accounts_server/{name}")
async def read_account_resource(name: str) -> str:
    account = Account.get(name.lower())
    return account.report()

if __name__ == "__main__":
    mcp.run(transport="stdio")

Hai điểm cần chốt: (1) tool wrapper rất mỏng, cho thấy MCP server nên reuse domain layer thay vì nhồi logic vào handler; (2) resources được address bằng URI pattern, giúp client đọc context theo tên tài khoản hoặc strategy.

Block 4 - accounts_client.py: handwritten client và schema bridge

Đây là block giàu giá trị học tập nhất của Day 2 vì nó lộ rõ protocol plumbing.

Purpose: tự list tools, call tool, read resource, và bridge sang OpenAI tools Key names: stdio_client, ClientSession, initialize(), FunctionTool 
async def list_accounts_tools():
    async with stdio_client(params) as streams:
        async with mcp.ClientSession(*streams) as session:
            await session.initialize()
            tools_result = await session.list_tools()
            return tools_result.tools

async def read_accounts_resource(name):
    async with stdio_client(params) as streams:
        async with mcp.ClientSession(*streams) as session:
            await session.initialize()
            result = await session.read_resource(f"accounts://accounts_server/{name}")
            return result.contents[0].text

async def get_accounts_tools_openai():
    openai_tools = []
    for tool in await list_accounts_tools():
        schema = {**tool.inputSchema, "additionalProperties": False}
        openai_tool = FunctionTool(
            name=tool.name,
            description=tool.description,
            params_json_schema=schema,
            on_invoke_tool=lambda ctx, args, toolname=tool.name:
                call_accounts_tool(toolname, json.loads(args))
        )
        openai_tools.append(openai_tool)
    return openai_tools

Đây là chỗ bạn thấy rõ 4 việc mà summary đã nhắc: initialize, list tools, call tool, và read resource. Thêm nữa, get_accounts_tools_openai() là một schema bridge - lớp nối schema giữa thế giới MCP và interface FunctionTool của OpenAI Agents SDK.

Section 9

Pitfalls - Lỗi / bẫy thường gặp

Failure modes - trường hợp dễ fail

  • Bọc mọi function thành MCP server: bị cuốn theo hype, quên hỏi có thật sự cần chia sẻ không.
  • Đặt logic vào wrapper thay vì domain: server nhìn “đúng giao thức” nhưng business layer yếu.
  • Quên session.initialize(): client lỗi vì lifecycle chưa sẵn sàng.
  • Dùng handwritten client như pattern mặc định: tăng chi phí code không cần thiết.
  • Nhầm schema MCP và schema OpenAI tools là một: wrapper invocation hoặc args shape bị sai.
  • Quên resources là primitive riêng: chỉ nghĩ MCP = tool calling.

Fix / prevention - cách tránh

  • Luôn hỏi trước: tool này cần shareability hay chỉ cần internal speed?
  • Giữ domain logic ở module riêng như accounts.py, để server wrapper mỏng.
  • Với client viết tay, coi initialize() là bước lifecycle bắt buộc.
  • Ưu tiên native SDK path khi framework đã hỗ trợ MCP đủ tốt.
  • Viết bridge rõ ràng như get_accounts_tools_openai() nếu cần compatibility.
  • Nhớ rằng resources có thể hữu ích cho context preload hoặc read-only workflows.

A: Shareability - khả năng chia sẻ, tức là biến capability của bạn thành thứ host hoặc agent khác dùng lại được theo chuẩn chung.

Q: Phần nào mới là giá trị thật trong Day 2, protocol hay business logic?

A: Business logic vẫn là giá trị thật; protocol chỉ là cách đóng gói và công bố logic đó ra ngoài.

Q: accounts_server.py minh họa thêm primitive nào ngoài tools?

A: Resources - tài nguyên, qua các URI như accounts://accounts_server/{name} và accounts://strategy/{name}.

Q: Vì sao handwritten client không phải lựa chọn mặc định?

A: Vì nó lặp lại nhiều plumbing mà SDK hiện đại đã hỗ trợ, nên chủ yếu có giá trị cho học tập, debug hoặc custom bridge.

Section 11

Quick Memory - Ôn nhanh cuối bài

Must remember - Phải nhớ

  1. MCP server có ý nghĩa nhất khi capability của bạn cần được chia sẻ và chuẩn hóa.
  2. accounts.py là domain layer; accounts_server.py là packaging layer.
  3. FastMCP cho phép expose cả tools và resources.
  4. mcp.run(transport="stdio") biến module server thành local MCP server spawn được.
  5. Native SDK path qua MCPServerStdio là đường mặc định gọn nhất cho OpenAI Agents SDK.
  6. accounts_client.py minh họa rõ lifecycle client: initialize, list, call, read.
  7. get_accounts_tools_openai() là ví dụ điển hình của schema bridge.
  8. Sau khi học internals, Day 2 khuyên bạn quay lại abstraction và ecosystem leverage thay vì tự viết plumbing mọi nơi.

Self-check - Tự kiểm tra

  • Bạn có phân biệt được “viết MCP server vì chia sẻ” với “viết MCP server vì hype” không?
  • Bạn có giải thích được vai trò riêng của accounts.py, accounts_server.py và accounts_client.py không?
  • Bạn có mô tả lại được vì sao initialize() là bước bắt buộc của client lifecycle không?
  • Bạn có nêu được khi nào nên chọn native SDK integration thay vì handwritten client không?
  • Bạn có giải thích được resource khác tool ở điểm nào bằng ví dụ Day 2 không?

Kết luận nhanh

Nếu Day 1 dạy bạn cách tận dụng MCP ecosystem, thì Day 2 dạy bạn điều kiện để chính mình đóng góp vào ecosystem đó. Ý quan trọng nhất cần mang sang Day 3 là: hãy hiểu plumbing đủ sâu để chọn abstraction đúng, chứ không phải để tái tạo plumbing ở mọi dự án.
