Core idea - Ý chính

Đóng gói ứng dụng nghiên cứu đa Agent thành một cấu trúc cơ sở mã nguồn dạng mô-đun (Modular Codebase). Dựng giao diện web tùy chỉnh bằng Gradio gr.Blocks, truyền luồng trạng thái thời gian thực thông qua Asynchronous Generators và asyncio.as_completed, sau đó đưa ứng dụng lên máy chủ đám mây Hugging Face Spaces.

Must know - Cần nhớ

  • Mô-đun hóa codebase giúp dễ bảo trì, unit test và phát triển các Agent độc lập.
  • Callback UI dùng async def ... yield giúp đẩy thông tin cập nhật từng bước, ngăn giao diện bị sập hoặc đóng băng (freeze).
  • asyncio.as_completed cho phép theo dõi và in ra tiến độ tác vụ song song mạng nhanh hơn asyncio.gather.
  • Bảo mật API keys qua cấu hình Settings Secrets của Hugging Face là bắt buộc khi deploy ứng dụng công khai.

Use in practice - Dùng khi nào

Áp dụng khi cần chuyển đổi các ý tưởng AI thử nghiệm thô thành ứng dụng SaaS web thực tế có giao diện thân thiện, hỗ trợ theo dõi tiến trình cào dữ liệu mạng động và chia sẻ cho người khác dùng thử một cách an toàn và miễn phí.

Core Theory - Lý thuyết cốt lõi

day5_summary.md

Modular Codebase & gr.Blocks

  • Modular Codebase - Cơ sở mã nguồn dạng mô-đun: Phương pháp chia nhỏ một ứng dụng cồng kềnh thành các file độc lập. Mỗi file đảm nhiệm duy nhất một chức năng riêng biệt. Cách thiết kế này giúp tối ưu hóa khả năng đọc hiểu code, hỗ trợ làm việc nhóm song song và viết unit tests độc lập.
  • gr.Blocks trong Gradio: Cú pháp dựng giao diện người dùng nâng cao cho phép tự do thiết lập bố cục (layout) và liên kết sự kiện (event binding) tùy chỉnh cho từng widget, khác biệt hoàn toàn với bố cục cứng nhắc của gr.Interface.

Asynchronous Generators & asyncio.as_completed

  • Asynchronous Generator - Bộ sinh bất đồng bộ: Hàm định nghĩa bằng async def và trả kết quả bằng từ khóa yield. Giúp máy chủ Python liên tục đẩy các bản cập nhật trạng thái trung gian về giao diện thông qua kết nối Server-Sent Events (SSE) thời gian thực mà không làm chặn (blocking) luồng xử lý chính.
  • asyncio.as_completed - Hoàn thành bất đồng bộ: Hàm trả về một bộ lặp (iterator) sinh ra các tác vụ ngay khi chúng hoàn thành, bất kể thứ tự kích hoạt ban đầu. Cơ chế này hoạt động theo nguyên lý "tác vụ mạng nào xong trước trả kết quả trước", giúp tính toán phần trăm tiến độ cào web linh hoạt.
Deep explanation - Giải thích sâu về asyncio.as_completed & Server-Sent Events (SSE)

Hiểu sâu hơn về luồng dữ liệu thời gian thực trong Deep Research App:

  • Khi người dùng click "Run" trên Gradio UI, Gradio backend đăng ký callback run(query). Vì callback này là một Asynchronous Generator, Gradio tự động mở một kết nối Server-Sent Events (SSE) để lắng nghe.
  • Bên trong ResearchManager().run(), tiến trình tìm kiếm song song kích hoạt 5 truy vấn cào web. Vòng lặp asyncio.as_completed được sử dụng. Mỗi khi một cuộc gọi API tìm kiếm Bing hoàn tất, vòng lặp hứng kết quả, tăng biến đếm num_completed và thực hiện yield f"Searching... {num_completed}/{total} completed".
  • Mỗi lệnh yield sẽ đóng gói chuỗi văn bản và gửi qua kênh SSE về trình duyệt người dùng. Gradio UI nhận thông điệp, cập nhật ngay lập tức DOM của thẻ Markdown mà không cần tải lại toàn bộ trang web. Điều này giải quyết triệt để lỗi "giao diện bị đóng băng" (UI freeze) khi chạy các tác vụ Agent tốn thời gian.

Workflow / Pipeline - Quy trình và luồng hoạt động

deep_research.py research_manager.py

UI & Manager Interaction Flow

  1. Gửi yêu cầu: Người dùng nhập câu hỏi vào query_textbox -> click `Run` -> kích hoạt callback run() bất đồng bộ.
  2. Ghi nhận vết chạy: ResearchManager sinh trace ID và yield link trace OpenAI lên giao diện.
  3. Lên kế hoạch: Gửi yêu cầu sang PlannerAgent để sinh 5 từ khóa tìm kiếm -> yield trạng thái lập kế hoạch hoàn tất.
  4. Cào web song song: Thực thi song song 5 truy vấn. Hứng kết quả bằng asyncio.as_completed -> yield tiến độ hoàn thành (ví dụ: `2/5 completed`) thời gian thực.
  5. Viết báo cáo: Gom dữ liệu thô gửi sang WriterAgent tổng hợp thành Markdown -> yield trạng thái.
  6. Gửi email & Hoàn tất: Gửi Markdown sang EmailAgent định dạng HTML và kích hoạt API SendGrid -> yield thông báo thành công và in báo cáo Markdown cuối cùng lên màn hình.

Techniques - Kỹ thuật sử dụng

Event Binding in gr.Blocks

Purpose: Đăng ký các hàm Python xử lý bất đồng bộ khi người dùng click nút bấm hoặc ấn Enter trên ô nhập liệu.

When to use: Khi dựng giao diện web bằng Blocks của Gradio.

Trade-off: Đòi hỏi nhà phát triển phải tự quản lý tham số inputs và outputs thủ công cho từng phần tử giao diện.

Progress Tracking via as_completed

Purpose: Cập nhật số lượng tác vụ hoàn thành động giúp người dùng biết hệ thống vẫn đang cào web bình thường.

When to use: Khi thực hiện nhiều cuộc gọi API mạng song song có thời gian phản hồi I/O khác nhau.

Trade-off: Logic lập trình phức tạp hơn và kết quả trả về không giữ nguyên thứ tự ban đầu.

Cloud Deployment via Gradio CLI

Purpose: Đẩy nhanh ứng dụng Python lên nền tảng đám mây Hugging Face Spaces để chia sẻ trực tuyến.

When to use: Khi cần xây dựng portfolio cá nhân hoặc chia sẻ demo sản phẩm cho cộng đồng.

Trade-off: Cần cấu hình thủ công khóa API nhạy cảm trong Settings Secrets để tránh lộ mã nguồn bảo mật.

# === deep_research.py ===
import gradio as gr
from dotenv import load_dotenv
from research_manager import ResearchManager

load_dotenv(override=True)

# Callback bất đồng bộ yield thông tin tiến trình cho Gradio UI
async def run(query: str):
    async for chunk in ResearchManager().run(query):
        yield chunk

# Thiết lập giao diện Blocks với theme sky thanh lịch
with gr.Blocks(theme=gr.themes.Default(primary_hue="sky")) as ui:
    gr.Markdown("# Deep Research")
    query_textbox = gr.Textbox(label="What topic would you like to research?")
    run_button = gr.Button("Run", variant="primary")
    report = gr.Markdown(label="Report")
    
    # Liên kết hành động click nút bấm và submit textbox với callback run
    run_button.click(fn=run, inputs=query_textbox, outputs=report)
    query_textbox.submit(fn=run, inputs=query_textbox, outputs=report)

if __name__ == "__main__":
    ui.launch(inbrowser=True)


# === research_manager.py ===
from agents import Runner, trace, gen_trace_id
from search_agent import search_agent
from planner_agent import planner_agent, WebSearchItem, WebSearchPlan
from writer_agent import writer_agent, ReportData
from email_agent import email_agent
import asyncio

class ResearchManager:
    async def run(self, query: str):
        # 1. Sinh trace ID ngẫu nhiên độc bản để liên kết log API
        trace_id = gen_trace_id()
        with trace("Research trace", trace_id=trace_id):
            # yield link trace lên UI ngay lập tức để người dùng tiện click giám sát
            yield f"View trace: https://platform.openai.com/traces/trace?trace_id={trace_id}"
            
            # 2. Lập kế hoạch tìm kiếm
            search_plan = await self.plan_searches(query)
            yield "Searches planned, starting to search..."     
            
            # 3. Duyệt tìm kiếm song song
            search_results = await self.perform_searches(search_plan)
            yield "Searches complete, writing report..."
            
            # 4. Viết báo cáo tổng hợp
            report = await self.write_report(query, search_results)
            yield "Report written, sending email..."
            
            # 5. Định dạng HTML và gửi thư qua SendGrid
            await self.send_email(report)
            yield "Email sent, research complete"
            yield report.markdown_report # yield kết quả Markdown cuối cùng lên UI

    async def plan_searches(self, query: str) -> WebSearchPlan:
        result = await Runner.run(planner_agent, f"Query: {query}")
        return result.final_output_as(WebSearchPlan)

    async def perform_searches(self, search_plan: WebSearchPlan) -> list[str]:
        num_completed = 0
        # Tạo các task bất đồng bộ
        tasks = [asyncio.create_task(self.search(item)) for item in search_plan.searches]
        results = []
        # as_completed giúp hứng kết quả tác vụ nào hoàn thành trước để cập nhật tiến độ
        for task in asyncio.as_completed(tasks):
            result = await task
            if result is not None:
                results.append(result)
            num_completed += 1
            print(f"Searching... {num_completed}/{len(tasks)} completed")
        return results

    async def search(self, item: WebSearchItem) -> str | None:
        input_data = f"Search term: {item.query}\nReason for searching: {item.reason}"
        try:
            result = await Runner.run(search_agent, input_data)
            return str(result.final_output)
        except Exception:
            return None # Tránh dừng tiến trình khi một truy vấn cào web bị lỗi

    async def write_report(self, query: str, search_results: list[str]) -> ReportData:
        input_data = f"Original query: {query}\nSummarized search results: {search_results}"
        result = await Runner.run(writer_agent, input_data)
        return result.final_output_as(ReportData)
    
    async def send_email(self, report: ReportData) -> None:
        await Runner.run(email_agent, report.markdown_report)
planner_agent.py, search_agent.py & writer_agent.py Framework: Pydantic Structured Outputs

Định nghĩa các lớp dữ liệu ràng buộc bằng Pydantic và khai báo cấu hình tham số độc lập cho từng Agent con. 

# === planner_agent.py ===
from pydantic import BaseModel, Field
from agents import Agent

HOW_MANY_SEARCHES = 5
INSTRUCTIONS = f"You are a helpful research assistant. Given a query, come up with a set of web searches \
to perform to best answer the query. Output {HOW_MANY_SEARCHES} terms to query for."

class WebSearchItem(BaseModel):
    # Khai báo reason trước query để LLM lập luận Chain of Thought trước khi gợi ý từ khóa
    reason: str = Field(description="Your reasoning for why this search is important to the query.")
    query: str = Field(description="The search term to use for the web search.")

class WebSearchPlan(BaseModel):
    searches: list[WebSearchItem] = Field(description="A list of web searches to perform to best answer the query.")
    
planner_agent = Agent(
    name="PlannerAgent",
    instructions=INSTRUCTIONS,
    model="gpt-4o-mini",
    output_type=WebSearchPlan, # Ép đầu ra cấu trúc JSON khớp schema WebSearchPlan
)


# === search_agent.py ===
from agents import Agent, WebSearchTool, ModelSettings

INSTRUCTIONS = (
    "You are a research assistant. Given a search term, you search the web for that term and "
    "produce a concise summary of the results. The summary must 2-3 paragraphs and less than 300 "
    "words. Capture the main points. Write succintly, no need to have complete sentences or good "
    "grammar. This will be consumed by someone synthesizing a report, so its vital you capture the "
    "essence and ignore any fluff. Do not include any additional commentary other than the summary itself."
)

search_agent = Agent(
    name="Search agent",
    instructions=INSTRUCTIONS,
    tools=[WebSearchTool(search_context_size="low")], # Tiết kiệm token đầu vào bằng context size low
    model="gpt-4o-mini",
    model_settings=ModelSettings(tool_choice="required"), # Ép Agent bắt buộc chạy công cụ tìm kiếm
)


# === writer_agent.py ===
from pydantic import BaseModel, Field
from agents import Agent

INSTRUCTIONS = (
    "You are a senior researcher tasked with writing a cohesive report for a research query. "
    "You will be provided with the original query, and some initial research done by a research assistant.\n"
    "You should first come up with an outline for the report that describes the structure and "
    "flow of the report. Then, generate the report and return that as your final output.\n"
    "The final output should be in markdown format, and it should be lengthy and detailed. Aim "
    "for 5-10 pages of content, at least 1000 words."
)

class ReportData(BaseModel):
    short_summary: str = Field(description="A short 2-3 sentence summary of the findings.")
    markdown_report: str = Field(description="The final report")
    follow_up_questions: list[str] = Field(description="Suggested topics to research further")

writer_agent = Agent(
    name="WriterAgent",
    instructions=INSTRUCTIONS,
    model="gpt-4o-mini",
    output_type=ReportData, # Ép đầu ra cấu trúc JSON khớp schema ReportData
)
email_agent.py Integration: SendGrid Function Tool

Tách rời logic gửi email thành một mô-đun riêng, đóng gói SendGrid Client làm công cụ gửi thư dạng HTML thông qua decorator @function_tool. 

# === email_agent.py ===
import os
from typing import Dict
import sendgrid
from sendgrid.helpers.mail import Email, Mail, Content, To
from agents import Agent, function_tool

# Đóng gói hàm Python thành công cụ của Agent thông qua @function_tool
@function_tool
def send_email(subject: str, html_body: str) -> Dict[str, str]:
    """Send an email with the given subject and HTML body"""
    # Khởi tạo SendGrid API Client
    sg = sendgrid.SendGridAPIClient(api_key=os.environ.get("SENDGRID_API_KEY"))
    from_email = Email("ed@edwarddonner.com")  # Bắt buộc thay bằng email gửi đã xác thực trên SendGrid
    to_email = To("ed.donner@gmail.com")      # Email người nhận báo cáo
    content = Content("text/html", html_body)
    mail = Mail(from_email, to_email, subject, content).get()
    
    response = sg.client.mail.send.post(request_body=mail)
    print("Email response status code:", response.status_code)
    return "success"

INSTRUCTIONS = """You are able to send a nicely formatted HTML email based on a detailed report.
You will be provided with a detailed report. You should use your tool to send one email, providing the 
report converted into clean, well presented HTML with an appropriate subject line."""

email_agent = Agent(
    name="Email agent",
    instructions=INSTRUCTIONS,
    tools=[send_email], # Đăng ký công cụ send_email cho Agent
    model="gpt-4o-mini",
)

Options / Trade-offs - Bản đồ lựa chọn

gr.Blocks Xây dựng ứng dụng web phức tạp, bố cục linh hoạt và luồng dữ liệu tùy chỉnh. Đòi hỏi viết nhiều dòng cấu hình UI thủ công, tăng độ phức tạp của codebase. Dễ gặp lỗi liên kết sự kiện (event binding) nếu đặt sai tham số widget.
gr.Interface Dựng nhanh các bản thử nghiệm chức năng (MVP) đơn giản chỉ trong vài dòng code. Giao diện cố định, không thể can thiệp sâu vào các sự kiện trung gian. Không hỗ trợ luồng stream trạng thái trung gian một cách tự nhiên.
asyncio.as_completed Ứng dụng web cần cập nhật phần trăm tiến độ cào web tức thời cho người dùng. Kết quả trả về không giữ nguyên thứ tự ban đầu của danh sách tác vụ. Cần quản lý lỗi ngoại lệ (exception) chặt chẽ trong từng tác vụ con.
asyncio.gather Tác vụ cần bảo toàn thứ tự kết quả trả về trùng khớp tuyệt đối với danh sách đầu vào. Phải đợi toàn bộ các tác vụ cào web xong xuôi mới xuất dữ liệu một lần duy nhất. Gây cảm giác giao diện bị đóng băng nếu chạy số lượng tìm kiếm lớn.
Hugging Face Spaces Demo dự án cá nhân, chia sẻ trực tuyến nhanh chóng và miễn phí. Máy chủ miễn phí chạy CPU yếu, mã nguồn ứng dụng công khai (public). Dễ bị rò rỉ API keys nếu không cấu hình an toàn trong Settings Secrets.
Section 9

Pitfalls - Lỗi / bẫy thường gặp và cách phòng tránh

Failure modes - Trường hợp dễ fail

  • Giao diện web bị đóng băng hoàn toàn: Gradio UI không hiển thị tiến trình cào web trung gian mà đứng im 30-60 giây rồi hiển thị báo cáo cuối cùng đột ngột.
  • Runtime Error sau khi deploy đám mây: Ứng dụng Gradio chạy tốt ở cục bộ nhưng báo lỗi đỏ ngay khi gửi câu hỏi trên Hugging Face Spaces.
  • Link trace báo lỗi 404 không tìm thấy: Người dùng click vào đường link trace được hiển thị trên UI nhưng OpenAI dashboard báo lỗi không tồn tại dữ liệu.

Fix / prevention - Cách tránh & Sửa

  • Sử dụng bộ sinh bất đồng bộ: Đảm bảo hàm callback UI khai báo dạng async def, sử dụng vòng lặp async for để hứng dữ liệu từ manager và trả kết quả trung gian bằng từ khóa yield.
  • Thiết lập Secrets variables: Sau khi chạy lệnh gradio deploy, truy cập ngay vào mục Settings -> Variables and secrets của Space để khai báo OPENAI_API_KEY và SENDGRID_API_KEY.
  • Xác minh tài khoản trace: Đảm bảo tài khoản OpenAI đang đăng nhập trên trình duyệt trùng khớp với tài khoản API key được ứng dụng sử dụng, và dự án/workspace được chọn tương ứng.
Section 10

Flashcards & Q&A - Thẻ ôn tập và câu hỏi nhanh

Q: Sự khác biệt chính giữa asyncio.gather và asyncio.as_completed trong xử lý tác vụ song song là gì?

A:
  • asyncio.gather: Chờ tất cả tác vụ hoàn thành rồi trả về toàn bộ kết quả một lần, bảo toàn đúng thứ tự danh sách đầu vào.
  • asyncio.as_completed: Trả về kết quả của từng tác vụ ngay khi nó hoàn tất (first-come, first-served), không giữ thứ tự, thích hợp cập nhật tiến độ UI thời gian thực.

Q: Tại sao cần sử dụng Asynchronous Generator (async def ... yield) cho callback của Gradio?

A: Vì tiến trình chạy của Agent cào web và viết báo cáo tốn nhiều thời gian. Sử dụng yield giúp backend liên tục đẩy các trạng thái cập nhật trung gian về giao diện người dùng thông qua kết nối Server-Sent Events (SSE), ngăn giao diện bị đóng băng.

Q: gr.Blocks khác gr.Interface như thế nào trong thiết kế Gradio UI?

A:
  • gr.Interface: Tự động tạo giao diện nhanh gọn với bố cục cố định, khó tùy biến luồng sự kiện.
  • gr.Blocks: Cho phép lập trình viên tự thiết kế bố cục giao diện phức tạp bằng HTML/CSS tùy chỉnh và can thiệp sâu vào đăng ký sự kiện của từng thành phần.

Q: Tại sao không được đẩy file .env chứa API keys lên Hugging Face Space khi deploy?

A: Vì Hugging Face Spaces mặc định công khai mã nguồn (public repo). Nếu đẩy file .env lên, API keys sẽ bị lộ công khai. Bạn bắt buộc phải khai báo API keys bảo mật trong phần cấu hình Settings Secrets của Space.
