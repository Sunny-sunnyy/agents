Cheat Sheet - Tóm tắt cốt lõi

Core idea - Ý chính

Lập trình bất đồng bộ (Async IO) sử dụng Event Loop để chạy các Coroutine trên một luồng duy nhất là nền tảng cốt lõi của các multi-agent frameworks như OpenAI Agents SDK, giúp tối ưu hóa tài nguyên mạng khi giao tiếp với LLM API.

Must know - Cần nhớ

  • Async IO không chạy đa luồng cấp OS mà chuyển đổi ngữ cảnh (context switching) cực nhẹ trên 1 thread.
  • Dùng async def định nghĩa coroutine, bắt buộc gọi bằng await để thực thi.
  • OpenAI Agents SDK có 3 lớp chính: Agent, Runner, trace.

Use in practice - Thực tế

Áp dụng ngay khi lập trình ứng dụng có nhiều Agent gọi API LLM đồng thời qua asyncio.gather, tránh tình trạng ứng dụng bị đơ (block) trong thời gian chờ đợi phản hồi mạng. Khi viết code, áp dụng Vibe Coding từng phần nhỏ để kiểm soát chất lượng sinh mã của LLM.

Mục 4

Core Theory - Lý thuyết cốt lõi

Docx LLM

Key concepts - Khái niệm chính

  • Asynchronous Python (async IO) - Lập trình bất đồng bộ: Mô hình lập trình cho phép thực hiện nhiều tác vụ đồng thời trên một luồng duy nhất mà không bị chặn (non-blocking), tối ưu hóa thời gian chờ phản hồi mạng.
  • Coroutine - Hàm bất đồng bộ đặc biệt: Đối tượng được tạo ra từ hàm khai báo bằng async def, có thể tạm dừng thực thi tại các điểm await và khôi phục trạng thái sau đó.
  • Event Loop - Vòng lặp sự kiện: Trình điều phối chạy ngầm liên tục để quản lý và phân bổ thời gian chạy cho các coroutine đang ở trạng thái sẵn sàng.
  • Non-opinionated framework - Framework không áp đặt: Triết lý thiết kế tối giản, cung cấp các khối gạch nền tảng và để nhà phát triển tự quyết định cấu trúc dự án và luồng điều khiển (ví dụ: OpenAI Agents SDK), trái ngược với LangChain.

Concept relationships - Mối quan hệ

Event Loop hoạt động như một trái tim trung tâm điều phối. Các Coroutine là các thực thể công việc được đăng ký vào Event Loop. Khi một coroutine gặp điểm chặn I/O (gọi API LLM qua mạng), nó sử dụng từ khóa await để tạm thời nhường quyền điều khiển (yield) lại cho Event Loop, giúp Event Loop lập tức chuyển sang chạy một coroutine khác trong hàng đợi mà không cần chờ đợi một cách vô ích.

Deep explanation - Giải thích sâu về Event Loop đơn luồng

Trong hệ điều hành, lập trình đa luồng (multithreading) truyền thống đòi hỏi OS phải liên tục thực hiện chuyển ngữ cảnh (context switch) ở cấp độ phần cứng. Việc này tiêu tốn nhiều tài nguyên bộ nhớ và có nguy cơ gặp lỗi tranh chấp dữ liệu (race conditions).

Ngược lại, Event Loop của Python hoạt động ở cấp độ ứng dụng trên một luồng duy nhất. Nó liên tục kiểm tra danh sách các tác vụ bất đồng bộ. Khi gặp lệnh await, Python sẽ lưu lại trạng thái thanh ghi và bộ nhớ của coroutine đó rồi chuyển sang chạy task khác. Khi hệ điều hành báo tác vụ I/O đã hoàn tất, coroutine cũ sẽ được đưa trở lại hàng đợi để tiếp tục chạy. Cơ chế này nhẹ hơn hàng trăm lần so với OS threads, cho phép xử lý hàng ngàn kết nối I/O cùng một lúc.

Mục 5

Workflow / Pipeline - Quy trình hoạt động

Docx LLM

Event Loop Task Dispatching - Quy trình Event Loop

  1. Đăng ký: Danh sách coroutines được nạp vào Event Loop (ví dụ qua asyncio.gather).
  2. Thực thi: Event Loop lấy coroutine đầu tiên ra chạy cho đến khi gặp điểm chặn await.
  3. Nhường quyền: Coroutine nhường quyền điều khiển, chuyển sang trạng thái chờ. Event Loop lập tức chạy coroutine tiếp theo.
  4. Hoàn thành: Khi I/O phản hồi, Event Loop xếp lịch đưa coroutine trở lại hàng đợi để chạy tiếp từ vị trí tạm dừng cho đến khi trả ra kết quả cuối cùng.

Agent Execution Lifecycle - Quy trình khởi chạy Agent

  1. Khởi tạo: Định nghĩa đối tượng Agent với các tham số name, instructions (system prompt) và model.
  2. Giám sát: Mở context manager with trace("Tên Trace"): để cấu hình thu thập dữ liệu telemetry.
  3. Thực thi: Gọi Runner.run(agent, prompt) kèm từ khóa await để khởi chạy vòng lặp xử lý bất đồng bộ, tự động xử lý các vòng lặp tool calling nếu có cho đến khi LLM kết thúc hội thoại.
Mục 6

Techniques - Kỹ thuật sử dụng

Parallel Coroutines with asyncio.gather

Purpose: Gom nhiều tác vụ bất đồng bộ độc lập lại và thực thi đồng thời nhằm tối ưu hóa tổng thời gian chờ đợi.

When to use: Khi cần thực hiện nhiều cuộc gọi API LLM độc lập cùng lúc (ví dụ hỏi ý kiến nhiều agent chuyên biệt).

Trade-off: Khó kiểm soát lỗi hơn; một coroutine con bị crash có thể làm ảnh hưởng đến kết quả của cả nhóm tác vụ.

Telemetry Tracing with with trace

Purpose: Ghi lại toàn bộ chuỗi sự kiện, tham số đầu vào/đầu ra và hoạt động gọi công cụ (tool call) của Agent để hiển thị trực quan.

When to use: Bắt buộc dùng trong quá trình phát triển để debug hành vi của Agent qua giao diện web.

Trade-off: Gửi dữ liệu telemetry lên cloud của OpenAI, yêu cầu kết nối mạng và có thể tăng nhẹ độ trễ cuộc gọi.

Incremental Vibe Coding

Purpose: Sinh mã nguồn an toàn với AI, tránh bẫy mã nguồn rác/lỗi (code slop) khi sinh quá nhiều code cùng lúc.

When to use: Khi xây dựng các tính năng mới hoặc làm việc với các thư viện/SDK mới chưa quen thuộc.

Trade-off: Đòi hỏi lập trình viên phải cực kỳ kỷ luật, chia nhỏ bài toán (10 dòng/lượt) và liên tục chạy kiểm thử thủ công.

Code Walkthrough - Phân tích mã nguồn

Code Docx LLM

Phân tích file thực hành: 1_lab1.ipynb

File notebook này hướng dẫn các bước cơ bản nhất để khởi tạo một Agent kể chuyện cười (Jokester) bằng OpenAI Agents SDK, thực thi bất đồng bộ trong context manager của Trace và cách lấy phản hồi cuối cùng.

Mục đích: Khởi tạo & chạy thử Agent cơ bản Lớp chính: Agent, Runner, trace Môi trường: Jupyter Notebook (.ipynb) 
# Bước 1: Nhập các lớp cốt lõi từ thư viện agents và thư viện cấu hình môi trường
from dotenv import load_dotenv
from agents import Agent, Runner, trace

# Bước 2: Nạp API Key của OpenAI từ file .env cục bộ
# Tham số override=True đảm bảo ghi đè các biến môi trường cũ trong hệ thống
load_dotenv(override=True)

# Bước 3: Định nghĩa thực thể Agent
# jokester là một tác nhân kể chuyện cười sử dụng mô hình gpt-4o-mini
agent = Agent(
    name="Jokester", 
    instructions="You are a joke teller", 
    model="gpt-4o-mini"
)

# Bước 4: Thực thi Agent bất đồng bộ bên trong khối Trace giám sát
# "Telling a joke" là nhãn hiển thị của trace trên OpenAI Platform (platform.openai.com/traces)
with trace("Telling a joke"):
    # Sử dụng từ khóa await trước Runner.run vì đây là một coroutine bất đồng bộ.
    # Trong môi trường Jupyter Notebook, tính năng Top-level Await được hỗ trợ mặc định.
    result = await Runner.run(agent, "Tell a joke about Autonomous AI Agents")
    
    # In ra nội dung phản hồi dạng chuỗi ký tự cuối cùng của Agent
    print(result.final_output)

Options / Trade-offs - So sánh lựa chọn

So sánh cơ chế xử lý đồng thời trong Python

Phương pháp Phù hợp nhất khi Đánh đổi (Trade-off) Rủi ro tiềm ẩn (Risk)
Multiprocessing (Đa tiến trình) Tác vụ tính toán nặng (CPU-bound) như xử lý ảnh, ma trận, AI model cục bộ. Chi phí tài nguyên bộ nhớ rất lớn do mỗi tiến trình có vùng nhớ riêng biệt. Giao tiếp giữa các tiến trình phức tạp, khởi tạo tiến trình chậm.
Multithreading (Đa luồng hệ điều hành) Tác vụ I/O bound ở mức độ trung bình khi làm việc với các thư viện cũ không hỗ trợ async. Tiêu tốn tài nguyên bộ nhớ cho mỗi thread, bị giới hạn bởi khóa GIL của Python. Dễ gặp lỗi tranh chấp bộ nhớ (race conditions), khó debug.
Async IO (Bất đồng bộ - Đơn luồng) Tác vụ I/O bound chiếm phần lớn thời gian (như gọi LLM API qua mạng). (Khuyên dùng) Không giải quyết được tác vụ tính toán CPU-bound. Code viết theo cú pháp phức tạp hơn. Một tác vụ CPU-bound nặng không nhường quyền (non-yielding) sẽ làm đơ toàn bộ ứng dụng.

So sánh trường phái xây dựng Agent Framework

Loại Framework Ví dụ Ưu điểm (Pros) Nhược điểm (Cons)
Opinionated (Áp đặt kiến trúc) LangChain, CrewAI Cung cấp sẵn các mẫu thiết kế (templates), nhiều tính năng ăn liền cho bài toán phổ thông. Rất cồng kềnh, khó tùy biến sâu, khó kiểm soát luồng chạy ngầm bên dưới.
Non-opinionated (Tự do kiến trúc) OpenAI Agents SDK Siêu nhẹ, dễ học, kiểm soát hoàn toàn luồng điều khiển (control flow), dễ dàng debug. (Khuyên dùng) Lập trình viên phải tự xây dựng các thành phần quản lý trạng thái khi hệ thống phình to.
Mục 9

Pitfalls - Các bẫy lỗi thường gặp

Failure modes - Các trường hợp lỗi

  • Quên từ khóa await: Gọi coroutine nhưng chương trình chỉ trả về một đối tượng dạng <coroutine object...> mà không thực thi logic bên trong hàm.
  • Lỗi cú pháp Await ở file Python (.py): Viết từ khóa await ở cấp ngoài cùng trong file script Python truyền thống dẫn đến lỗi SyntaxError: 'await' outside function.
  • Không cập nhật được biến môi trường: Đã thay đổi API Key trong file .env nhưng code chạy vẫn nhận giá trị cũ của hệ thống.

Fix / prevention - Cách khắc phục

  • Sửa lỗi quên await: Luôn viết cú pháp nhận kết quả dạng result = await my_async_function() đối với các coroutine.
  • Sửa lỗi SyntaxError ngoài Jupyter: Bọc toàn bộ logic chạy vào trong hàm bất đồng bộ async def main(): và gọi chạy Event Loop thông qua asyncio.run(main()).
  • Sửa lỗi nạp env: Sử dụng tham số ghi đè khi nạp biến môi trường: load_dotenv(override=True) ngay dòng đầu chương trình.

Flashcards & Quiz - Ôn tập tương tác

Flashcards (Nhấn vào nút để xem đáp án)

Q: Coroutine trong Python là gì?

A: Là đối tượng được tạo ra từ hàm khai báo bằng async def, có khả năng tạm ngưng thực thi tại các điểm await để nhường luồng chạy cho tác vụ khác và được đánh thức lại bởi Event Loop khi tác vụ hoàn thành.

Q: Handoff trong Multi-agent hoạt động ra sao?

A: Hoạt động bằng cách cho phép một Agent trả về thực thể của một Agent khác từ bên trong một hàm công cụ (tool function). Runner sẽ nhận biết và chuyển giao toàn bộ ngữ cảnh trò chuyện sang Agent mới.

Q: Lợi thế của non-opinionated framework là gì?

A: Giúp tối giản hóa mã nguồn boilerplate, cho phép lập trình viên toàn quyền kiểm soát luồng điều khiển, dễ dàng tùy biến cấu trúc và tích hợp linh hoạt mà không bị gò bó bởi các lớp trừu tượng phức tạp.
