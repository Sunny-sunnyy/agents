Core idea - Ý chính

Tận dụng hạ tầng đa mô hình (Gemini, DeepSeek, Groq) bằng cách bọc client bất đồng bộ qua OpenAIChatCompletionsModel. Must know - Cần nhớ

  • SDK hỗ trợ đa mô hình qua các endpoint tương thích với client AsyncOpenAI của hãng.
  • Tham số output_type gán BaseModel Pydantic ép LLM trả định dạng JSON chuẩn.
  • Đặt tripwire_triggered=True trong GuardrailFunctionOutput để ném exception dừng hệ thống ngay lập tức.

Use in practice - Thực tế

Áp dụng khi triển khai các hệ thống Agent thương mại (production) tiếp nhận thông tin trực tiếp từ người dùng. Thiết lập bộ lọc Input Guardrail độc lập để lọc bỏ dữ liệu tên riêng, thông tin bảo mật trước khi gửi lên mô hình ngôn ngữ chính, tránh rò rỉ dữ liệu hoặc prompt injection.

Mục 4

Core Theory - Lý thuyết cốt lõi

Docx LLM

Key concepts - Khái niệm chính

  • OpenAI-compatible Endpoints - Điểm cuối tương thích OpenAI: Địa chỉ API bên thứ ba (như Google, DeepSeek, Groq) được xây dựng giống hệt chuẩn API của OpenAI về cấu trúc đường dẫn và định dạng dữ liệu truyền tải.
  • OpenAIChatCompletionsModel: Lớp bọc mô hình trong SDK giúp liên kết model name cụ thể với custom client chỉ định.
  • Structured Outputs - Đầu ra có cấu trúc: Cơ chế bắt buộc LLM phản hồi tuân thủ nghiêm ngặt định dạng JSON được mô tả qua Pydantic BaseModel để hệ thống tự động parse dữ liệu an toàn.
  • Tripwire - Bẫy dây / Thanh chắn bảo mật: Cờ hiệu boolean trong guardrail, nếu mang giá trị True sẽ kích hoạt dừng luồng chạy ngay lập tức.
  • PII (Personally Identifiable Information) - Thông tin nhận dạng cá nhân: Các thông tin nhạy cảm của người dùng (tên riêng, số điện thoại) cần được bảo mật và chặn lọc qua guardrail.

Concept relationships - Mối quan hệ

Mối quan hệ an toàn: Khi có yêu cầu, SDK chạy danh sách input_guardrails đầu tiên. Bộ lọc guardrail này chạy một guardrail_agent chuyên trách có tham số output_type ép kiểu Pydantic. Kết quả phân tích trả về cờ tripwire_triggered. Nếu sập bẫy, hệ thống dừng lập tức bằng Exception và hoàn toàn không khởi chạy Agent chính hay gọi các công cụ viết thư nháp chạy trên đa mô hình (Gemini, DeepSeek, Groq).

Deep explanation - Tại sao Guardrails độc lập lại bảo mật tốt hơn Prompting?

1. Hạn chế của Prompting (Chỉ dẫn hệ thống):

Việc viết câu lệnh cấm như "Không được tiết lộ thông tin cá nhân của người dùng" vào system prompt của Agent chính không đảm bảo an toàn tuyệt đối. LLM có tính chất không xác định (non-deterministic) và rất dễ bị vượt qua bằng kỹ thuật Prompt Injection hoặc Jailbreak (người dùng cố tình lừa Agent bỏ qua chỉ thị ban đầu). Khi đó Agent chính vẫn sẽ thực thi công cụ gửi thư và làm lộ thông tin nhạy cảm.

2. Sức mạnh của Guardrails (Thanh chắn độc lập):

Guardrail hoạt động như một lớp phần mềm bọc ngoài chạy độc lập. Khi người dùng truyền prompt vào, hàm guardrail sẽ được gọi trước tiên. Nó sử dụng một Guardrail Agent nhỏ để phân loại xem prompt có chứa PII hay không dưới dạng Structured Outputs (Pydantic). Nếu phát hiện vi phạm, hàm guardrail trả về cờ tripwire_triggered=True. SDK nhận diện cờ này và lập tức ném ra Exception chặn đứng luồng thực thi, không cho phép Agent chính hay bất kỳ công cụ gửi mail nào được kích hoạt. Chốt chặn lập trình cứng này đảm bảo an toàn 100% cho ứng dụng doanh nghiệp.

Workflow / Pipeline - Luồng hoạt động

Docx LLM

Input Guardrail Execution - Quy trình chạy Guardrail

  1. Nhận Prompt: Người dùng nhập tin nhắn gửi cho Careful Sales Manager (ví dụ: "Send email from Alice").
  2. Chạy Guardrail: SDK tự động gọi hàm guardrail_against_name trước tiên.
  3. Gọi Guardrail Agent: Hàm chạy Name check agent để phân tích tin nhắn và nhận về đối tượng Pydantic NameCheckOutput.
  4. Kiểm tra: Nếu thuộc tính is_name_in_message == True, trả về GuardrailFunctionOutput(tripwire_triggered=True).
  5. Chặn đứng: SDK nhận cờ tripwire, dừng lập tức hệ thống và ném Exception. Các Agent bán hàng con hoàn toàn không được gọi.

Multi-Model SDR Pipeline - Luồng Automated SDR đa mô hình

  1. Manager nhận việc: Sales Manager (chạy GPT-4o-mini) tiếp nhận yêu cầu gửi thư.
  2. Gọi đa mô hình: Manager gọi song song 3 công cụ:
    • sales_agent1 chạy mô hình deepseek-chat qua DeepSeek client.
    • sales_agent2 chạy mô hình gemini-2.0-flash qua Gemini client.
    • sales_agent3 chạy mô hình llama-3.3-70b-versatile qua Groq client.
  3. Lựa chọn & Handoff: Manager tự so sánh 3 bản nháp email và Handoff bản nháp tốt nhất sang cho Email Manager gửi đi.
Mục 6

Techniques - Kỹ thuật sử dụng

Custom OpenAI Client (AsyncOpenAI)

Purpose: Cấu hình trỏ địa chỉ API và nạp khóa xác thực của các bên thứ ba để tích hợp đa mô hình vào SDK.

When to use: Khi muốn gọi mô hình của Google Gemini, DeepSeek hoặc Groq Llama.

Trade-off: Đòi hỏi lập trình viên phải tự đăng ký tài khoản, nạp tiền và quản lý nhiều API Keys khác nhau trong file .env.

Pydantic Structured Outputs

Purpose: Ép kiểu trả về của Agent tuân thủ nghiêm ngặt schema dữ liệu để hệ thống tự động xử lý an toàn bằng code.

When to use: Phù hợp khi xây dựng các tác nhân phân loại, trích xuất dữ liệu, chấm điểm hoặc guardrails.

Trade-off: LLM phải hỗ trợ Structured Outputs và có thể tiêu tốn thêm thời gian suy luận để định dạng định dạng JSON đầu ra.

Input Guardrail Decorator (@input_guardrail)

Purpose: Đăng ký hàm kiểm duyệt an toàn đầu vào, tự động chạy trước khi kích hoạt Agent chính và các công cụ con.

When to use: Sử dụng cho các Agent tiếp nhận prompt tự do từ người dùng trong môi trường thực tế.

Trade-off: Làm tăng nhẹ thời gian thực thi tổng thể do phải chạy thêm cuộc gọi LLM kiểm duyệt phụ trợ.

# Khai báo địa chỉ Base URL tương thích OpenAI của Google, DeepSeek và Groq
GEMINI_BASE_URL = "https://generativelanguage.googleapis.com/v1beta/openai/"
DEEPSEEK_BASE_URL = "https://api.deepseek.com/v1"
GROQ_BASE_URL = "https://api.groq.com/openai/v1"

# Khởi tạo các client bất đồng bộ AsyncOpenAI với API Key tương ứng
deepseek_client = AsyncOpenAI(base_url=DEEPSEEK_BASE_URL, api_key=deepseek_api_key)
gemini_client = AsyncOpenAI(base_url=GEMINI_BASE_URL, api_key=google_api_key)
groq_client = AsyncOpenAI(base_url=GROQ_BASE_URL, api_key=groq_api_key)

# Bọc các client và khai báo tên mô hình thông qua OpenAIChatCompletionsModel
deepseek_model = OpenAIChatCompletionsModel(model="deepseek-chat", openai_client=deepseek_client)
gemini_model = OpenAIChatCompletionsModel(model="gemini-2.0-flash", openai_client=gemini_client)
llama3_3_model = OpenAIChatCompletionsModel(model="llama-3.3-70b-versatile", openai_client=groq_client)

# Gán đối tượng model vừa bọc vào tham số model khi khai báo Agent

2. Ép kiểu đầu ra Structured Outputs bằng Pydantic BaseModel:

from pydantic import BaseModel

# Định nghĩa schema cấu trúc đầu ra mong muốn
class NameCheckOutput(BaseModel):
    is_name_in_message: bool
    name: str

# Khởi tạo guardrail agent và truyền class NameCheckOutput vào tham số output_type
guardrail_agent = Agent( 
    name="Name check",
    instructions="Check if the user is including someone's personal name in what they want you to do.",
    output_type=NameCheckOutput, # Bắt buộc đầu ra tuân thủ schema JSON của Pydantic
    model="gpt-4o-mini"
)

3. Viết hàm Input Guardrail kiểm duyệt và bắt cờ Tripwire:

from agents import input_guardrail, GuardrailFunctionOutput

# Đăng ký hàm Python làm Input Guardrail chạy trước
@input_guardrail
async def guardrail_against_name(ctx, agent, message):
    # Chạy guardrail agent để phân tích tin nhắn đầu vào
    result = await Runner.run(guardrail_agent, message, context=ctx.context)
    
    # Do sử dụng Structured Outputs, ta truy cập trực tiếp thuộc tính boolean đã parse
    is_name_in_message = result.final_output.is_name_in_message
    
    # Trả về đối tượng GuardrailFunctionOutput.
    # tripwire_triggered nhận giá trị boolean; nếu True sẽ chặn đứng chương trình.
    return GuardrailFunctionOutput(
        output_info={"found_name": result.final_output},
        tripwire_triggered=is_name_in_message
    )

4. Khởi chạy và bắt Exception bảo vệ an toàn bằng try-except:

careful_sales_manager = Agent(
    name="Sales Manager",
    instructions=sales_manager_instructions,
    tools=tools,
    handoffs=[emailer_agent],
    model="gpt-4o-mini",
    input_guardrails=[guardrail_against_name] # Đăng ký bộ lọc bảo mật đầu vào
)

# Kịch bản vi phạm: tin nhắn chứa tên riêng "Alice"
message = "Send out a cold sales email addressed to Dear CEO from Alice"

# Sử dụng try-except để bắt exception khi guardrail sập bẫy, tránh crash chương trình
try:
    with trace("Protected Automated SDR"):
        result = await Runner.run(careful_sales_manager, message)
except Exception as e:
    # In ra thông báo lỗi tripwire của guardrail
    print(f"Chặn đứng thành công! Guardrail phát hiện vi phạm: {e}")

So sánh Structured Outputs và Free Text Outputs

Định dạng Ưu điểm (Pros) Nhược điểm (Cons) Khuyên dùng khi nào
Structured Outputs (Pydantic) Đầu ra được ép kiểu JSON chuẩn, tự động parse thành đối tượng Python có thuộc tính rõ ràng. Lập trình an toàn, tin cậy. (Khuyên dùng) Mô hình LLM phải hỗ trợ Structured Outputs và tốn thêm thời gian suy luận để format JSON. Các tác vụ trích xuất thực thể, phân loại, chấm điểm hoặc các bộ lọc bảo mật.
Free Text Outputs (Văn bản tự do) Thực thi nhanh hơn, mô hình LLM tự do diễn đạt nội dung phong phú và tự nhiên. Khó parse dữ liệu bằng code Python; dễ gặp lỗi định dạng hoặc lỗi cú pháp khi dùng regex. Các tác vụ viết email nháp, viết truyện, hội thoại tự do với người dùng.
Mục 9

Pitfalls - Các bẫy lỗi thường gặp

Failure modes - Các trường hợp lỗi

  • Lỗi kết nối đa mô hình (AuthenticationError): Ném ra ngoại lệ xác thực ngay khi chạy Agent đa mô hình do thiếu khai báo API Key tương ứng trong file .env.
  • Lỗi định dạng Base URL: Khởi tạo client AsyncOpenAI thành công nhưng bị báo lỗi API Connection Error khi gọi chạy mô hình do viết sai Base URL của nhà cung cấp.
  • Crash hệ thống do Guardrail: Khi Input Guardrail phát hiện vi phạm và sập bẫy tripwire, toàn bộ ứng dụng Python bị dừng đột ngột (crash) và báo lỗi hệ thống.

Fix / prevention - Cách khắc phục

  • Xử lý thiếu API key: [Alert] Bắt buộc cấu hình đầy đủ các keys trong `.env`. Nếu thiếu key của hãng nào, tạm thời cấu hình gán model="gpt-4o-mini" cho Agent đó để test.
  • Cú pháp Base URL chuẩn: Viết chính xác cấu trúc API URL của các hãng, ví dụ DeepSeek: https://api.deepseek.com/v1 (phải có `/v1` ở cuối).
  • Bọc khối try-except: Luôn bọc lời gọi chạy Agent chính bằng khối lệnh try-except để bắt ngoại lệ do guardrail ném ra và in thông báo thân thiện.

Flashcards & Quiz - Ôn tập tương tác

Flashcards (Nhấn vào nút để xem đáp án)

Q: Khác biệt khi truyền model dạng string so với truyền dạng đối tượng model?

A: Truyền dạng chuỗi (string) sẽ khiến Agent tự động sử dụng client mặc định trỏ về máy chủ OpenAI, trong khi truyền đối tượng OpenAIChatCompletionsModel cho phép tùy biến trỏ sang các Custom Clients (Base URLs) của hãng khác.

Q: Tại sao Structured Outputs giúp Agent tự trị chạy ổn định hơn?

A: Vì nó ép buộc LLM trả về cấu trúc JSON chặt chẽ và an toàn, loại bỏ hoàn toàn các lỗi cú pháp khi sinh văn bản tự do, giúp Agent Manager dễ dàng đưa ra quyết định dừng chính xác dựa trên thuộc tính đối tượng rõ ràng.

Q: Tripwire (Bẫy dây) hoạt động như thế nào trong Guardrail?

A: Hoạt động bằng cách gán cờ tripwire_triggered=True trong đối tượng GuardrailFunctionOutput trả về. Khi nhận diện cờ này, SDK lập tức dừng hệ thống và ném ra Exception chặn đứng Agent chính.

  3. Input Guardrail được khai báo bằng decorator @input_guardrail và tự động chạy trước Agent chính.
  4. Cờ tripwire_triggered=True sẽ kích hoạt lỗi chặn đứng chương trình và ném Exception bảo vệ.
  5. Bắt buộc dùng khối lệnh try-except khi gọi chạy Agent có guardrail để tránh sập hệ thống.
